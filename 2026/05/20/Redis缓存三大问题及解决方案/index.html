Redis 缓存三大问题及解决方案

在”黑马点评”项目中引入 Redis 缓存后，高并发场景下暴露了三个经典问题。本文从问题本质、实际踩坑到最终方案，完整记录这条学习链路。

一、为什么用缓存

项目最初商户查询是纯查 MySQL，QPS 一高数据库就扛不住。引入 Cache-Aside 模式后：

响应时间从几十毫秒降到 1 毫秒以内，吞吐量立刻上去了。

但缓存是好东西，用不好会引出三个经典问题。

二、缓存穿透

2.1 什么是缓存穿透

查询一个数据库永远不存在的数据，Redis 没有，MySQL 也没有——每次请求都打到数据库。

实际遇到的问题：恶意用不存在的店铺 id 刷接口，MySQL 不停执行 SELECT * FROM tb_shop WHERE id=99999，全都返回 null，缓存完全没起作用。

2.2 解决方案：空值缓存

数据库查不到时，往 Redis 存一个空字符串占位 ""，给一个短 TTL，这样下一次相同请求在 Redis 层就直接返回 null，不再穿透到 DB。

2.3 落地代码

参数 值 说明

实际踩坑：最初忘记做空值缓存，攻击者随便用不存在的 id 就能把数据库压垮。加上之后，相同的无效请求全部在 Redis 层被挡掉。

三、缓存击穿

3.1 什么是缓存击穿

单个热点 key 过期的一瞬间，大量并发请求同时打到数据库。

实际遇到的问题：爆款店铺缓存刚好过期，同一毫秒来了几百个请求，全部穿透到 MySQL，数据库 CPU 瞬间飙到 90%。

3.2 预防和穿透的区别

问题 攻击目标 触发条件
穿透 不存在的数据 请求不存在的 key
击穿 存在的热点数据 热点 key 过期瞬间

3.3 方案一：互斥锁

核心思路：缓存未命中时，只让一个线程去查库重建，其他线程等待。

3.4 落地代码

关键设计点：

  1. while 循环重试：不使用递归，避免栈溢出风险
  2. DoubleCheck：拿到锁后再次查 Redis，可能前一个持有锁的线程已经重建好了
  3. 锁粒度：lock:shop:{id}，按店铺 ID 隔离，不同店铺不互相阻塞
  4. 锁 TTL 10 秒：防止拿到锁的线程异常退出导致死锁

3.5 方案二：逻辑过期

核心思路：key 物理上永不过期，value 里存一个逻辑过期时间。读到过期数据时，先返回旧数据，后台异步重建。

四、缓存雪崩

4.1 什么是缓存雪崩

大量 key 在同一时刻过期，或者 Redis 整个宕机，所有请求瞬间打到数据库，数据库可能直接挂掉。

4.2 解决方案

多级缓存 本地 Caffeine + Redis，Redis 挂时本地还能扛

六、Jackson LocalDateTime 序列化踩坑

6.1 遇到的问题

店铺 Shop 实体里包含 LocalDateTime 字段（如 openHours），直接用 new ObjectMapper() 序列化到 Redis 时报错：

6.2 原因

Jackson 默认只认识 java.util.Date，不认识 Java 8 的 LocalDateTime，需要显式注册 JavaTimeModule。

6.3 修复

disable(WRITE_DATES_AS_TIMESTAMPS) 让时间以 ISO 字符串格式存储，避免序列化为时间戳数组。后来统一使用 Hutool 的 JSONUtil 替代了 Jackson，这个问题彻底消除。

七、总结

问题 症状 根因 方案 落地
穿透 无效请求打穿 DB 缓存空值未处理 空值缓存 "" 短 TTL CacheClient 内置
击穿 热点过期 DB 过载 并发查库无互斥 互斥锁 / 逻辑过期 queryWithMutex / queryWithLogicalExpire
雪崩 批量过期 / Redis 宕 缓存集体失效 TTL 随机化 / 集群 预留方案

三道防线协同保护数据库，配合 CacheClient 工具类封装，后续新模块接入缓存只需一行代码。

个人心得：缓存不是简单的”查缓存 → 没查到就查库 → 放进去”，穿透、击穿、雪崩每一个都能在高并发下暴露致命问题。本质是容量规划 + 并发控制 + 容错兜底三个维度的综合设计。空值缓存是容量层（减少无效 DB 流量），互斥锁是并发层（控制 DB 访问并发度），TTL 随机化是容错层（分散失效时间避免集中冲击）。
