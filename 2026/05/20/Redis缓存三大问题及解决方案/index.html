Redis 缓存三大问题及解决方案

在”黑马点评”项目中为商户查询引入 Redis 缓存后，高并发场景下三个经典问题逐一暴露。本文从问题本质、实际踩坑、代码演进到最终架构，完整复盘这条学习链路。

一、项目背景

1.1 业务场景

黑马点评是一个类大众点评的位置服务平台，商户模块需要提供店铺详情查询。数据库表 tb_shop 结构如下：

三、缓存穿透

3.1 什么是缓存穿透

攻击者故意查询一个数据库永远不存在的数据。 Redis 没缓存，MySQL 也查不到，每次都穿透到数据库。

3.2 项目中实际出现的场景

攻击者用不存在的店铺 id 狂刷 /shop/{id} 接口：

每个请求都白白执行了一次 SELECT * FROM tb_shop WHERE id=99999，MySQL 连接很快被打满。

3.3 解决方案：空值缓存

数据库查不到时，往 Redis 存一个空字符串占位 ""，给一个短 TTL（2分钟）。下一次相同 id 的请求查 Redis 时，命中空字符串，直接返回 null，不再访问数据库。

3.5 关键设计

设计点 说明
空值用什么存 空字符串 ""
为什么不用 null Redis 不支持存 null
为什么 TTL 只有 2 分钟 防止长期占位，同时足够挡掉批量攻击
为什么不用布隆过滤器 空值缓存更简单，布隆需要额外维护一套索引

3.6 扩展：布隆过滤器

空值缓存的缺点是无法区分”这个 id 不存在”和”这个 id 可能存在但缓存过期了”。布隆过滤器能解决这个问题，但需要额外维护。

当前项目优先使用空值缓存方案，布隆过滤器可作为后续升级方向。

四、缓存击穿

4.1 什么是缓存击穿

某个热点 key 过期的一瞬间，大量并发请求同时打到数据库。

和穿透的区别：

问题 查询的数据 触发条件
穿透 数据根本不存在 一直不存在
击穿 数据真实存在且是热点 key 过期的那一瞬间

4.2 项目中实际出现的场景

某爆款店铺 id=7 的缓存刚好在 30 分钟 TTL 后过期。同一毫秒内，数个请求同时发现 Redis 里没有缓存，全部去查 MySQL：

数百个线程同时执行相同的 SELECT，数据库 CPU 瞬间飙高。

4.3 问题根源

在 Redis 的查询和 MySQL 的查询之间，没有任何互斥机制。所有线程同时发现缓存缺失，同时去查库。

4.4 方案一：互斥锁

核心思路：缓存未命中时，用 Redis setnx 命令实现分布式互斥锁，只让一个线程去查库重建缓存，其他线程休眠等待。

Redis 的 SET key value NX EX timeout 命令：

如果 key 已存在（锁被别人持有了），命令返回 (nil)，即获取锁失败。

tryLock 实现

锁的设计要点

设计点 值 原因
锁的 TTL 10 秒 防止拿到锁的线程挂了导致死锁，Redis 自动过期兜底
锁的粒度 lock:shop:{id} 按店铺 ID 隔离，不同店铺不互相阻塞
锁的 value 线程标识 解锁时可以校验”是不是我的锁”，防止误删

为什么不用递归

第一版用递归实现重试，获取锁失败后递归调用自己等待：

但递归有两个隐患：

  1. 栈溢出：锁 TTL 10 秒 ÷ 休眠 50ms = 最多 200 层递归，虽然理论上不会溢出，但不够安全
  2. 锁释放逻辑分散：try-finally 跨越多层递归，代码理解成本高

DoubleCheck 是什么

拿到锁之后，第一步不是急着查库，而是再查一次 Redis。原因：可能前一个持有锁的线程刚重建完缓存释放了锁，当前线程拿到了锁但缓存里已经有数据了。

4.5 方案二：逻辑过期

核心思想

互斥锁方案需要等待——锁被别人占了就得 sleep。逻辑过期的思路是完全不等待：key 在 Redis 里永不过期，读到旧数据先返回，后台异步重建。

RedisData 包装类

Redis key 的物理 TTL 不设置（或设得极长），判断过期全靠 expireTime。

互斥锁 逻辑过期
核心目标 数据一致性优先 可用性优先
是否等待 等待（sleep） 不等待
数据新鲜度 总是最新的 可能短暂返回旧数据
实现复杂度 较简单 较复杂，需要 RedisData 包装 + 线程池
缓存预热 不需要 需要提前 setWithLogicalExpire
性能 等待时需反复查 Redis 零等待，极致性能
适用场景 一般业务 超高并发热点

当前项目默认启用互斥锁方案。逻辑过期方案的完整代码保留在项目中，需要时可立刻切换。

五、缓存雪崩

5.1 什么是缓存雪崩

大量 key 在同一时刻集中过期，或者 Redis 整个宕机，所有请求瞬间打到数据库，数据库可能直接挂掉。

5.2 第一种情况：批量 key 同时过期

在项目中，如果管理员一次性导入了 100 个热门店铺数据，它们的缓存 TTL 都是 30 分钟。30 分钟后，这 100 个 key 在同一秒内同时过期，查询这些店铺的请求全部穿透到 MySQL。

解决方案：TTL 加随机偏移

效果：30 分钟的 TTL 分散到 20~40 分钟之间，100 个 key 的过期时间被均匀散开。

5.3 第二种情况：Redis 宕机

整个 Redis 挂掉，所有缓存都不可用，请求全打到 MySQL。

解决方案

方案 说明
Redis 主从 + 哨兵 Redis 本身高可用，不单点故障
多级缓存 本地 Caffeine + 远程 Redis，Redis 挂了本地缓存还能扛
熔断降级 Hystrix/Sentinel，DB 压力过大时限流
异地多活 多数据中心，高级方案

Redis 不可用时，至少业务还能通过直接查库继续运行（虽然会慢很多）。

七、Jackson LocalDateTime 序列化踩坑

7.1 问题现象

最初用 Jackson 的 ObjectMapper 序列化 Shop 对象存 Redis 时，遇到错误：

7.2 原因

Shop 实体包含了 LocalDateTime 类型的字段（如 createTime、updateTime、openHours）。Jackson 默认只认识 java.util.Date，不认识 Java 8 引入的 LocalDateTime。

7.3 修复

显式注册 JavaTimeModule：

如果 write_dates_as_timestamps 不关，LocalDateTime 会被序列化为 [2026, 5, 16, 9, 43, 31] 这样的数组，反序列化时容易出错。关掉后以 "2026-05-16T09:43:31" 这样的 ISO-8601 标准字符串存储。

7.4 后续改进

后来统一使用 Hutool 的 JSONUtil 替代 Jackson，JSONUtil.toJsonStr() 和 JSONUtil.toBean() 内部已经处理好了序列化问题，不再需要手动配置 ObjectMapper。

八、总结

8.1 问题 — 方案 — 落地对照

问题 症状 根因 方案 项目落地
穿透 无效请求直击 DB 没缓存空值 "" 占位，短 TTL 2min CacheClient.queryWithMutex 内置
击穿 热点过期 DB 瞬时过载 无并发查库控制 互斥锁 / 逻辑过期 queryWithMutex 启用 → queryWithLogicalExpire 备用
雪崩 批量过期 → DB 压垮 缓存集体失效 TTL 随机化 + Redis 集群 TTL 随机偏移（预留）

8.2 三道防线架构

8.3 思维链

缓存的本质不是简单的”查 → 没查到就放进去”。三个问题分别对应系统设计的三个维度：

  • 穿透 = 容量问题：无效请求占用了本该留给有效请求的 DB 连接池
  • 击穿 = 并发问题：没有互斥控制的热点数据访问是灾难性的
  • 雪崩 = 容错问题：没有防御式设计的系统在集中失效面前毫无还手之力

三者共同构成 Redis 缓存的完整防护体系。
