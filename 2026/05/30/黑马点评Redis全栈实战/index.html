黑马点评 Redis 全栈实战

在一个完整的点评类项目中，从缓存三大问题出发，逐步深入分布式锁、秒杀消息队列、社交关系、地理位置、签到统计、UV 去重——覆盖 Redis 全部九种数据结构的实战应用。

一、项目概览

黑马点评是一个类似大众点评的本地生活平台，基于 Spring Boot 2.7 + MyBatis-Plus 3.4 + Redis 5.0。核心模块包括：

  • 商户查询与缓存
  • 优惠券秒杀
  • 博客发布与社交（点赞、关注、Feed 流）
  • 附近商铺
  • 用户签到
  • UV 统计

这篇文章按照 Redis 数据结构的维度，逐一梳理每个模块的实现方案、Redis 命令、踩坑记录。

二、String — 商户缓存

2.1 基础缓存策略：Cache-Aside

2.2 缓存穿透

问题：大量请求查询不存在的 ID（如 id=-1），每次都穿透到 DB。

方案：空值缓存。数据库查不到时，把空字符串 "" 写入 Redis，TTL 设置 2 分钟。下次同样的请求直接返回 null，不查 DB。

2.3 缓存击穿

问题：热点 key 过期瞬间，大量并发打到 DB。

方案一：互斥锁

方案二：逻辑过期

物理 key 永远不过期，value 包一层 RedisData（含 expireTime）。读取时判断是否过期：

方案 适用场景 缺点
互斥锁 一致性要求高 拿不到锁会阻塞等待
逻辑过期 高可用，数据可以短暂不一致 返回的是旧数据

2.4 缓存雪崩

问题：大量 key 同时过期，或 Redis 宕机。

方案：TTL 加随机值打散 + Redis 集群 + 多级缓存。

2.5 踩坑：Jackson 序列化

Spring Boot 默认用 JDK 序列化，存到 Redis 里是一串 \xAC\xED 二进制，不可读。

解决：使用 StringRedisTemplate（自动用 String 序列化），手动 JSON 转换。

三、Lua + String — 分布式锁

3.1 从 JVM 锁到分布式锁

synchronized 是 JVM 级别的对象监视器，一个 Java 进程一把锁。启动两个 Spring Boot 实例后，两把锁互不认识，一人一单被打破。

根因：需要一个 JVM 外部、所有实例共享的”中间人”来管锁 → Redis。

3.2 自研 SimpleRedisLock

加锁：

  • NX = key 不存在才成功（互斥）
  • EX 5 = 5 秒自动过期（防死锁）
  • value = UUID + 线程名（标识锁的持有者）

解锁：不能直接 DEL——如果锁过期被 Redis 删了，另一个线程拿到锁，直接 DEL 会误删别人的锁。

用 Lua 把判断 + 删除包成原子操作：

Redis 单线程执行 Lua 脚本，GET → 判断 → DEL 一气呵成。

踩坑：myId.equals(threadId) 空指针——锁过期后 myId 为 null。反过来写 VALUE_ID_PREFIX.equals(myId) 就解决了（String.equals(null) 返回 false）。

3.3 SimpleRedisLock 的局限

问题 说明
固定 TTL 锁设 5 秒，业务跑 6 秒锁就丢了
不可重入 同一线程不能两次获取同一把锁
无锁重试 拿锁失败只能 sleep 轮询
无自动续期 没有看门狗

3.4 Redisson：工业级分布式锁

可重入：用 Redis Hash 存锁

重入：HINCRBY lock key field 1。释放：HINCRBY -1，归零才 DEL。

看门狗：后台定时任务每 10 秒 PEXPIRE 续期到 30 秒，只要线程活着锁就不会过期。

锁重试：Redis Pub/Sub 替代轮询

MultiLock vs RedLock：

  • MultiLock：同时向多个 Redis 节点加锁，全部成功才算拿到
  • RedLock：放宽到超过半数节点成功即可（两组多数派必然有交集）

十一、踩坑总结

问题 原因 解决
Field 'id' doesn't have a default value MyBatis-Plus IdType.AUTO 但表用雪花 ID 改为 IdType.INPUT
超卖（乐观锁未生效） if (stock < 1) 和 UPDATE 不是原子 UPDATE WHERE stock > 0
SimpleRedisLock NPE 解锁 锁过期后 myId 为 null String.equals(null) 返回 false
秒杀 Lua tonumber(nil) <= 0 绕过 Lua 中 tonumber(false) = nil，nil <= 0 = false (tonumber(...) or 0) <= 0
Stream API 不兼容 Spring Data Redis 2.3.x 缺少 claim() 等 API Lua 脚本 + 短轮询替代
QueryChainWrapper.remove() 不存在 MyBatis-Plus 3.4.3 不支持 baseMapper.delete(new QueryWrapper<>())
bitCount() / BITFIELD 报错 Spring Data Redis 版本 API 差异 execute(RedisCallback) 回调 + GETBIT 逐天查

十二、学习心得

这个项目最大的收获不是”会用 Redis”，而是理解了每种数据结构最适合什么场景：

  • 为什么点赞用 ZSet 而不是 Set？因为要按时间排序。
  • 为什么关注用 Set？因为要 SINTER 求交集。
  • 为什么 Feed 用推模式？因为粉丝少时推比拉快。
  • 为什么 UV 用 HyperLogLog 而不是 Set？因为内存差了一万倍。
  • 为什么 GEO 比 MySQL 算距离快？因为 Geohash 把二维降到一维，O(log N) vs O(N)。

每一步方案升级都不是凭空出现的，都是前一步的缺陷倒逼出来的。
