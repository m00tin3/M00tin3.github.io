这段代码在单线程下单没有问题，但并发场景下暴露了两个关键缺陷：

问题一：超卖。两个线程同时读到 stock=1，都通过了第 3 步的 if (seckillVoucher.getStock() < 1) 判断，然后都执行第 4 步的 UPDATE。最终一个扣到 0，另一个扣到 -1——库存出现负数，多卖了一件商品。

问题二：一人多单。同一个用户快速连续点击两次提交按钮，两个请求同时到达，都通过了库存检查，都创建了订单。结果一个用户买到了两件秒杀商品。

1.3 根本原因

这两个问题的本质都是读和写之间没有原子性保证。线程 A 读到库存值之后、更新库存之前，线程 B 也读了同一个旧值。

二、第一层防线：乐观锁防超卖

2.1 乐观锁和悲观锁的区别

在解决超卖问题之前，先理解两种并发控制思路：

悲观锁：假设冲突一定会发生，先加锁再操作。比如 SELECT ... FOR UPDATE，查询时就把行锁住，其他线程的查询被阻塞。

优点：绝对安全，不会出错。
缺点：性能差。每个 SELECT 都要排队，高并发下数据库连接全被阻塞。

乐观锁：假设冲突大概率不会发生，先操作，提交时再检查有没有别人改过。如果改过就重试或失败。

优点：无锁等待，并发性能极高。
缺点：冲突频繁时需要重试，适合”读多写少”场景。

秒杀场景中库存递减是单向操作（只减不增），非常适合用乐观锁。

2.5 优缺点

优点：

  • 实现极简，一行 gt("stock", 0) 搞定
  • 无额外字段（不需要 version 列）
  • 无锁等待，数据库不需要维护锁表
  • affected_rows=0 直接返回”库存不足”，业务语义精确

缺点：

  • 只能表达”够不够”（> 0），不能表达更复杂的更新条件
  • 如果业务需要知道”到底是被谁抢先了”，乐观锁做不到，得用悲观锁或分布式锁
  • 冲突频繁时（库存还剩 1 件，1000 人抢），大量 UPDATE 的 affected_rows=0，浪费数据库 IO

2.6 适用场景

乐观锁最适合秒杀、抢购这类库存单向递减 + 冲突概率随库存降低而逐渐增大的场景。库存充足时冲突少，绝大部分 UPDATE 都成功；库存快完时冲突多，但此时也快卖完了。

如果业务需要”先到先得并且严格排队”，或者需要”抢不到就阻塞等待”，那就更适合用悲观锁或 Redis 分布式锁。

四、分布式锁：从零造轮子

4.1 分布式锁的核心思路

找一个 JVM 外部、所有实例都能访问的”中间人”来管锁。所有实例都向这个中间人申请锁，它说”给你”你才能干活。Redis 就是理想的中间人：

  • 单线程执行命令：天然串行，不会有”两个同时拿到锁”的问题
  • 所有实例通过网络接入同一个 Redis：跨 JVM 共享
  • **原子命令 SETNX**：SET if Not eXists，一步完成”检查是否存在 + 设置值”

4.3 SimpleRedisLock 实现

4.3.1 加锁原理

Redis 的 SET key value NX EX timeout 命令包含了三个关键信息：

NX（Not eXists）：只有当 key 不存在时才能创建成功。如果 key 已经存在，返回 (nil)——这就是”锁已被别人持有”。

EX（Expire）：设置 key 的生存时间。如果拿到锁的线程挂了，key 也会在 5 秒后自动删除，不会造成死锁。

Value 存线程标识：解锁时需要判断”这把锁是不是我的”。存一个全局唯一的字符串（UUID + 线程名），每个实例的 UUID 前缀不同，同一实例内的线程名不同。

4.3.2 解锁的陷阱

最直接的思路——直接 DEL：

问题：线程 A 拿锁执行业务，5 秒到了锁自动过期被 Redis 删除。线程 B 此刻拿到了锁。线程 A 执行完去 unlock()——把线程 B 的锁删了。线程 C 又能拿锁，此时 B 和 C 同时认为自己持锁。

解决方案：解锁前先判断锁是不是自己的。

新的问题：第 1 步和第 3 步之间存在竞态窗口。

GET 和 DEL 之间不是原子操作。Redis 可能在中间插入了过期删除和线程 B 的 SET。

Lua 是 Redis 内置的微型脚本语言。可以把一段包含多个 Redis 命令的 Lua 代码发给 Redis，Redis 当成一个原子命令执行完。

4.5 SimpleRedisLock 的优缺点

优点：

  • 跨 JVM 互斥，完美替代 synchronized
  • 基于标准 Redis 命令（SETNX + EX），兼容所有 Redis 版本
  • Lua 脚本原子解锁，不误删
  • 代码量小，容易理解

缺点：

5.3 看门狗（Watchdog）自动续期

SimpleRedisLock 的痛点：必须预估业务执行时间，设一个足够大的固定 TTL。设短了业务没跑完锁就丢了，设长了死锁场景下锁存活太久。

Redisson 的解决方案：加锁成功后，额外启动一个后台定时任务（Watchdog），每隔 10 秒刷新一次锁的过期时间，只要持有锁的线程还活着，锁就永远不会过期。

JVM 崩溃怎么办？ 看门狗和业务线程在同一个 JVM 内。JVM 崩溃 → 看门狗线程也挂了 → 不再续期 → 30 秒后锁自动过期 → 其他线程可以拿到锁。所以死锁最多持续 30 秒。

不启用看门狗的情况：如果调用 tryLock 时手动指定了 leaseTime（锁持有时间），看门狗就不启动：

什么时候需要手动指定？当你明确知道业务执行的最长时间，且希望即使没调用 unlock 锁也能自动过期。

5.4 锁重试机制

SimpleRedisLock 的重试：sleep(50) + while 循环，每次都发一次 Redis 请求去试——如果 100 个线程在等待，每秒就是 100 × 20 = 2000 次无意义的网络请求。

Redisson 的优化：用 Redis Pub/Sub（发布/订阅）机制。

整个过程线程 A 只发了两次 Redis 请求（订阅 + 重新获取），不是每秒 20 次轮询。

如果线程 A 等了指定时间（如 1 秒）还没收到通知，就超时返回 false。

5.6 MultiLock 与 RedLock（简述）

MultiLock：同时向多个独立 Redis 节点加锁，全部成功才算成功。

使用场景：Redis 主从异步复制可能导致锁丢失——主节点刚写入锁数据就宕机，数据还没同步到从节点，从节点升主后锁没了。MultiLock 用多个独立 Redis 节点规避单点风险。

RedLock：MultiLock 的放宽版，超过半数节点成功即可。原理是”两个多数派必然有交集，交集节点上同一把锁只能属于一方”。

对于当前单机 Redis 项目，RLock 完全够用。

六、异步削峰：从 BlockingQueue 到 Redis Stream

6.1 为什么要异步

前面解决了超卖和一人一单，但吞吐量还有瓶颈。同步流程是：

Tomcat 线程从头到尾被占用，高并发下 200 个线程很快耗尽，新请求开始排队。但用户真正关心的是”抢到没有”，不需要等”写DB完成”。

异步思路：主线程干快活（Lua 校验），慢活（写DB）丢给后台。

6.2 第一版：JVM BlockingQueue

性能提升：Tomcat 线程立即释放，响应时间从 ~200ms 降到 ~1ms，吞吐量从 200 并发涨到 5000+。

致命问题：

无确认机制 take() 取走即删，消费者处理中途崩溃 → 订单丢失

6.6 消费者组

消费者组是 Stream 区别于其他 Redis 队列的杀手锏功能。

关键特性：

  • 一条消息只派发给组内一个消费者，不会重复派发
  • 多个消费者竞争消费，天然负载均衡
  • 消费者下线不影响，消息仍在 Stream 里
  • 每个消费者维护自己的”读到哪了”游标

6.7 PEL（Pending Entries List）

消费者拿到消息后到 XACK 之间，消息的状态是”已派发但未确认”，这个状态记录在 PEL 里。
