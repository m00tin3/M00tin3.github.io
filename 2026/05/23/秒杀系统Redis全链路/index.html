三、第二层防线：一人一单

库存问题解决了，但还有另一个问题：一个用户狂点提交，同一个券买好几次。

3.1 第一版：synchronized

userId.toString().intern() 的含义：toString() 每次返回新的 String 对象（不同引用），intern() 统一指向 JVM 字符串常量池中的同一个引用——这样才能当锁对象用。

3.2 问题暴露

启动两个 Spring Boot 实例后，同一个用户发两个请求被 Nginx 负载均衡到不同的 Tomcat，两个实例各有一把 synchronized 锁（JVM 对象监视器），互不认识，一人一单被打破。

根因：synchronized 是 JVM 级别的锁，一个 Java 进程一个。

加锁：基于 Redis SET key value NX EX timeout 命令

解锁：不能直接 DEL——如果锁已经过期被 Redis 删了，另一个线程拿到了锁，直接 DEL 会误删别人的锁。

需要三步：GET 校验是不是自己的锁 → 是才 DEL。但 GET 和 DEL 不是原子操作，中间可能刚好锁过期。

解决方案：Lua 脚本把判断 + 删除包成一个原子操作。

Redis 单线程执行 Lua 脚本，GET → 判断 → DEL 一气呵成。

5.3 看门狗（Watchdog）

Redisson 加锁后启动一个后台定时任务，每 10 秒刷新锁的过期时间：

只要持有锁的线程还活着，锁就不会过期。 这彻底解决了 SimpleRedisLock 固定 TTL 导致业务超时丢锁的问题。

注意：如果 tryLock 里指定了 leaseTime（锁持有时间），看门狗就不启动，由你手动控制。

5.4 锁重试

SimpleRedisLock 的重试是 sleep(50) + 循环，每次重试都是一次 Redis 网络请求，浪费资源。

Redisson 用 Redis 的 Pub/Sub：

不轮询、不浪费 CPU，锁释放后即时唤醒。

5.5 MultiLock 与 RedLock

MultiLock：同时向多个独立 Redis 节点加锁，全部成功才算成功。防止单点 Redis 宕机导致锁丢失。

RedLock：MultiLock 的放宽版，超过半数节点成功即可。容错更高，原理是”两组多数派必然有交集，交集节点上同一把锁只能属于一方”。

当前项目单机 Redis 用 RLock 足够，MultiLock 和 RedLock 是生产集群环境下的进一步兜底。

6.2 Stream 是什么

Stream 是 Redis 5.0 引入的类 Kafka 消息队列。可以理解为追加日志 + 消费者组。

每条消息有唯一 ID（格式 时间戳-序号），新消息追加到末尾。

每条消息只分配给组内一个消费者。两个实例的消费线程加入同一个消费者组，共享消息队列。

6.5 PEL（Pending Entries List）

消费者拿到消息后还没 XACK 的消息都在 PEL 里。消费者中途崩溃了，消息不会被丢——下次启动时从 PEL 捞回来重试。

失败重试 无 PEL + XCLAIM
JVM 崩溃 数据全丢 消息在 Redis 里
