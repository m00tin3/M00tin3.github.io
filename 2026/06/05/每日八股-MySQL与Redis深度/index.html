1. MySQL 索引的数据结构及各自优势

MySQL 常见的四种索引数据结构：

数据结构 MySQL 中使用 核心优势
B+ 树 InnoDB、MyISAM 默认 支持范围查询、排序、矮胖 IO 少
哈希 MEMORY 引擎、InnoDB 自适应哈希 等值查询 O(1)
全文索引（倒排索引） InnoDB FULLTEXT（5.6+） 文本搜索：MATCH...AGAINST
空间索引（R 树） MyISAM、InnoDB（5.7+）GIS 地理位置查询：ST_Contains

一、B+ 树 —— 主力，InnoDB 默认

优势 说明
多叉矮胖 一个节点 = 一个磁盘页(16KB)，能存几百个 key → 3 层能索引千万行
叶子有序 + 双向链表 范围查询 BETWEEN 100 AND 200 找到起点顺着链表走
自平衡 节点分裂/合并，始终 O(log n)
稳定 增删改都 O(log n)，不会退化

二、哈希索引 —— 等值查询 O(1)，但有硬伤

优势 劣势
✅ 等值查询 O(1)，极快 ❌ 不支持范围查询
✅ 索引结构紧凑 ❌ 不支持排序（ORDER BY 不走索引）
❌ 哈希冲突多时性能退化
❌ 不支持部分索引匹配（LIKE 'ab%' 不走）
❌ InnoDB 不支持手动建，只有自适应哈希（自动为热点页建）

三、全文索引（倒排索引）—— 文本搜索专用

优势 劣势
✅ 文本搜索极快 ❌ 只适用于文本
✅ 支持相关度排序、布尔搜索 ❌ 中文需要分词（ngram / IK 分词器）
✅ 比 LIKE '%xxx%' 快 N 倍 ❌ 只在 InnoDB 5.6+ 支持

四、R 树（空间索引）—— 地理位置专用

五、四种结构横向对比

对比 B+ 树 哈希 全文索引 R 树
等值查询 ✅ O(log n) ✅ O(1) ❌ ✅
范围查询 ✅ 最佳 ❌ 不行 ❌ ✅（空间范围）
排序 ✅ ❌ ❌ ❌
模糊搜索 ✅ LIKE 'abc%' ❌ ✅ MATCH...AGAINST ❌
存储引擎 InnoDB/MyISAM MEMORY（手动） InnoDB/MyISAM InnoDB/MyISAM
核心场景 一切的基础 等值快速查找 文本搜索 地图 / GIS
问题 答案
InnoDB 默认索引结构？ B+ 树，聚簇索引叶子存整行数据
哈希索引为什么不常用？ 不支持范围查询和排序，只适合等值
InnoDB 能用哈希索引吗？ 不能手动建，但自适应哈希索引（AHI）会自动为热点数据建
什么叫自适应哈希？ InnoDB 观察到某些页被频繁访问 → 自动在内存建哈希 → 等值查询加速
全文索引和 LIKE 的区别？ 全文建立倒排索引，LIKE 是全表扫

2. 索引设计原则

原则 说明 反例
1. WHERE/JOIN/ORDER BY 字段建索引 查询最频繁的列优先 不加索引 → 全表扫
2. 选区分度高的列 值越分散索引效果越好 性别(男/女) → 区分度只有 2 → 索引无效
3. 联合索引遵循最左前缀 (a,b,c) → a、ab、abc 走索引，bc 不走 跳过 a 直接查 b → 索引失效
4. 不要过度建索引 索引占空间，增删改都要维护 一个小表建 10 个索引 → 写入变慢
5. 尽量用覆盖索引 查询字段全在索引里 → 不回表 SELECT * FROM t WHERE name='x' → 即使 name 有索引也要回表
6. 避免在索引列上使用函数 WHERE DATE(create_time)='2026-01-01' 不走索引 改为 WHERE create_time >= '2026-01-01' AND create_time < '2026-01-02'
7. 避免隐式类型转换 WHERE phone = 13800138000（phone 是 varchar）→ 不走索引 写成 WHERE phone = '13800138000'
8. 尽量用自增主键 新数据永远追加到最后 → 不分裂 UUID → 随机插入 → 页分裂频繁
9. 区分度公式 COUNT(DISTINCT col)/COUNT(*) 接近 1 才有效 区分度 < 0.1 基本没效果

最左前缀原则详解：

排查索引是否生效：EXPLAIN SELECT ... → 看 key 列（是否用上索引）、type 列（ALL=全表扫不好，ref/range=好）。

问题 答案
什么字段不该建索引？ 区分度低（性别）、频繁修改、很小的表（几百行）
联合索引为什么有最左前缀？ B+ 树按字段顺序组织，跳过头列就无法定位
函数为什么导致索引失效？ INDEX(col) 存的是原始值，函数改变了值 → 无法匹配
怎么知道索引是否生效？ EXPLAIN → type=ALL 没走，Using index=覆盖

3. MySQL 锁的类型

MySQL 的锁按三个维度划分：锁粒度、锁模式、加锁方式。

一、按粒度

锁 粒度 并发度 说明
表锁 整张表 最低 MyISAM 默认，InnoDB 特殊场景也用（如没有索引的更新）
行锁 一行记录 最高 InnoDB 的默认级，对索引记录加锁
页锁 一个数据页 居中 BDB 引擎使用，基本见不到

二、行锁的三种模式（InnoDB）

锁 符号 行为 互斥
Record Lock 锁住索引记录 锁住单个 key 写写互斥，读写看隔离级别
Gap Lock 锁住索引记录之间的间隙 锁住间隙，不允许插入新数据 不同事务的 Gap 不互斥
Next-Key Lock Record + Gap 合体 锁住行 + 行前面的间隙 默认行锁形式

Gap Lock 的核心作用：解决幻读。MySQL 的 REPEATABLE READ 能防幻读，靠的就是 Gap Lock。

三、按加锁方式（共享锁 vs 排他锁）

锁 也叫 行为 加锁方式
共享锁（S 锁） 读锁 允许别人读，不允许别人写 SELECT ... LOCK IN SHARE MODE
排他锁（X 锁） 写锁 不允许别人读，也不允许别人写 SELECT ... FOR UPDATE / DML
意向锁（IS/IX） 表级标志 加行锁前先在表上做个标记，便于快速判 自动加

S 锁和 X 锁的兼容矩阵：

S X IS IX
S ✅ ❌ ✅ ❌
X ❌ ❌ ❌ ❌
IS ✅ ❌ ✅ ✅
IX ❌ ❌ ✅ ✅

四、按锁定范围（乐观锁 vs 悲观锁）

悲观锁 乐观锁
理念 别人会抢，先锁了再说 大概率没人抢，更新时再检查
实现 SELECT ... FOR UPDATE UPDATE ... SET version=version+1 WHERE version=?
适用 冲突多的场景（库存扣减） 冲突少的场景（改个人资料）
性能 锁等待，吞吐低 无锁，但冲突多时重试开销大

五、死锁

避免死锁：① 访问顺序一致 ② 批量操作排序 ③ 拆分大事务 ④ innodb_lock_wait_timeout 设定超时

问题 答案
MyISAM 和 InnoDB 的锁区别？ MyISAM 只有表锁；InnoDB 支持行锁
行锁锁的是什么？ 索引记录（不是数据行本身，没有索引会锁全表！）
Next-Key Lock 解决什么问题？ 幻读（Gap 禁止插入，Repeatable Read 级别生效）
乐观锁和悲观锁怎么选？ 冲突多 → 悲观；冲突少 → 乐观

4. MySQL 的执行计划（EXPLAIN）怎么看

一句话：EXPLAIN SQL 告诉你 MySQL 准备怎么执行这条 SQL——走哪个索引、扫多少行、有没有回表，是 SQL 优化的第一工具。

一、核心字段速查表

字段 看什么 理想值 问题值
type 访问方式（最重要！） const/eq_ref/ref/range **ALL**（全表扫描）
key 实际使用的索引 有值（索引名） NULL（没用索引）
rows 预估扫描行数 越小越好 几十万/百万 → 可能要加索引
Extra 额外信息 Using index（覆盖） Using filesort/Using temporary
key_len 索引使用长度 越大说明联合索引用到的列越多 —
possible_keys 可选索引 — NULL（没有索引可用）

二、最关键的 type（访问类型）

从快到慢：

type 含义 触发条件
const 常量查询，1 行 WHERE id = 1（主键）
eq_ref 关联查询中，驱动表一行匹配被驱动表一行 JOIN 用主键/唯一键关联
ref 非唯一索引等值匹配 WHERE name = '张三'（name 有索引）
range 索引范围扫描 WHERE age > 20 / BETWEEN / IN
index 全索引扫描（扫整个索引树） 索引能覆盖但无法定位 → 比 ALL 好，但仍慢
ALL ❌ 全表扫描 没走任何索引 → 必须优化！

三、Extra 必须关注的三个

Extra 含义 好还是坏
Using index ✅ 覆盖索引，不回表 好
Using filesort ❌ 额外排序（没用到索引排序） 坏，加索引或优化 ORDER BY
Using temporary ❌ 用了临时表（排序/去重/GROUP BY） 最坏，必须加索引

问题 答案
type=ALL 必须优化吗？ 是，除非表只有几百行
Using filesort 怎么解决？ ORDER BY 列加索引，且保证最左前缀
Using temporary 怎么解决？ GROUP BY 列加索引
rows 是不是实际扫行数？ 不是，是估算值，大致反映工作量

5. 事务的基本特性（ACID）和隔离级别

一、事务四大特性（ACID）

特性 含义 实现方式（InnoDB）
A 原子性 要么全做，要么全不做 undo log（回滚日志）：失败了根据 undolog 撤回
C 一致性 事务前后数据完整性不变 A+I+D 共同保证
I 隔离性 多个事务互不干扰 锁（MVCC + 行锁 + Gap Lock）
D 持久性 提交后数据永久保存 redo log（重做日志）：崩溃恢复

二、并发事务的三大问题

问题 描述 本质
脏读 读到别人未提交的数据（可能被回滚） 读了不该读的
不可重复读 同一事务内两次读同一条数据，结果不一致（别人 update 了） 同一个数据变了
幻读 同一事务内两次查，条数不一样（别人 insert/delete 了） 行数变了

三、四种隔离级别

隔离级别 脏读 不可重复读 幻读 性能
READ UNCOMMITTED ⚠️ ⚠️ ⚠️ 最高
READ COMMITTED ✅ ⚠️ ⚠️ 高
REPEATABLE READ（MySQL 默认） ✅ ✅ ✅（Next-Key Lock） 中
SERIALIZABLE ✅ ✅ ✅ 最差

四、MVCC 如何实现隔离（不用锁也能保证一致性）

MVCC（多版本并发控制）核心：**每个事务看到一个”快照”**。

概念 说明
版本链 每行数据有 trx_id（最后修改的事务 ID）+ roll_pointer（指向 undo log），形成历史版本链
ReadView 事务启动时记录：当前活跃事务列表、最小/最大事务 ID
可见性规则 对比行版本的事务 ID 和 ReadView：
→ 小于最小活跃 ID = 已提交 ✅ 可见
→ 在活跃列表中 = 未提交 ❌ 不可见
→ 大于最大活跃 ID = 未来事务 ❌ 不可见
问题 答案
ACID 各通过什么实现？ undo log(原子)、redo log(持久)、锁+MVCC(隔离)
MySQL 默认隔离级别？ REPEATABLE READ，Oracle 是 READ COMMITTED
MVCC 解决什么问题？ 读写不冲突，不用锁也能保证一致性
RR 怎么防幻读？ Next-Key Lock（行锁 + 间隙锁），MVCC 的快照读也能避免

7. ACID 靠什么保证

特性 依靠 原理一句话
A 原子性 undo log（回滚日志） 记录数据修改前的旧值，失败时拿旧值覆盖回来
C 一致性 A + I + D 共同保证 undo + redo + 锁机制，外加业务约束（外键、触发器）
I 隔离性 MVCC + 锁 MVCC 提供快照读不冲突 + 锁（行锁/Gap/Next-Key）控制写
D 持久性 redo log（重做日志） 先写日志再写磁盘，崩溃后用 redo log 恢复提交过的数据

一、原子性 —— undo log（回滚日志）

关键点 说明
记录什么 修改前的旧值（逻辑日志，反操作）
存在哪 undo 表空间，innodb_undo_directory 指定
有什么用 回滚 + MVCC 读取历史版本

二、持久性 —— redo log（重做日志）

关键点 说明
记录什么 物理层面的修改：”哪个表空间、哪个页、偏移量、改了什么”
为什么快 顺序追加写，不像数据页是随机 IO
两个文件 ib_logfile0 + ib_logfile1，循环写入
刷盘策略 innodb_flush_log_at_trx_commit：0=每秒刷，1=每次提交刷(最安全)，2=写到 OS 缓存

redo log vs undo log 对比

redo log undo log
记录内容 物理修改（改了什么） 逻辑反操作（旧值）
作用 崩溃恢复（持久性） 回滚 + MVCC（原子性）
写方式 顺序追加，循环写 随机写
何时产生 数据修改时 数据修改前
何时删除 循环覆盖 事务提交后不会立即删除，MVCC 需要

三、隔离性 —— MVCC + 锁

快照读（普通 SELECT）：靠 MVCC，读历史版本，不用锁

当前读（SELECT … FOR UPDATE / UPDATE / DELETE）：靠 Next-Key Lock

四、数据修改的完整流程（redo + undo 配合）

问题 答案
undo log 除了回滚还有什么用？ MVCC 读取历史版本（如果没 undo log，RR 隔离级别无法看到快照）
redo log 为什么不直接写磁盘？ 数据页是随机 IO，redo log 是顺序追加写，快很多
事务提交时数据一定落盘了吗？ 不一定！redo log 落盘就算提交成功，数据页异步刷盘
innodb_flush_log_at_trx_commit=1 的作用？ 每次提交强制刷 redo log，最安全，但性能有影响

8. 什么是 MVCC

一句话：MVCC（多版本并发控制）让读不阻塞写、写不阻塞读，通过给每行数据维护多个历史版本，每个事务看到一个属于自己的”快照”。

二、MVCC 的四个核心组件

组件 存在哪 作用
隐藏列 trx_id 每行数据 记录最后一次修改本行的事务 ID（6 字节）
隐藏列 roll_pointer 每行数据 指向 undo log 中的旧版本（7 字节）

ReadView 内存 事务启动时记录当前活跃事务的快照，判断历史版本是否可见

五、ReadView 生成的时机决定了隔离级别

隔离级别 ReadView 生成时机 效果
REPEATABLE READ 事务第一次查询时创建，之后复用 整个事务看到同一个快照 → 可重复读
READ COMMITTED 每次查询都重建新的 ReadView 每次看到最新已提交数据 → 不可重复读

六、快照读 vs 当前读

类型 SQL 用的技术 读什么
快照读 SELECT ...（不加锁的查询） MVCC（ReadView + undo log） 历史版本（快照）
当前读 SELECT ... FOR UPDATE、UPDATE、DELETE、INSERT Next-Key Lock 最新版本（且会加锁）

七、MVCC 解决了什么？

问题 靠什么
读写互不阻塞 快照读用 MVCC 读旧版本 → 不需要等写锁释放
可重复读 RR 级别下 ReadView 创建后复用 → 同一事务看到同一快照
读已提交 RC 级别每次重建 ReadView → 看到最新已提交

MVCC 不能解决写冲突——两个事务同时改同一行，还是要等锁。它的核心价值是让读不被写阻塞。

问题 答案
MVCC 的核心组件？ 隐藏列(trx_id/roll_pointer) + undo log 版本链 + ReadView
ReadView 是什么？ 事务启动时的”快照”，记录哪些事务活跃，用来判断版本可见性
RR 和 RC 在 MVCC 上的区别？ RR 一次 ReadView 用整个事务，RC 每次查询重建
快照读和当前读的区别？ 快照读看历史版本(不加锁)，当前读看最新版本(加锁)

10. MySQL 索引类型对数据库性能的影响

索引不是免费的，建得不对反而拖慢性能。从四个维度分析：

一、B+ 树索引（最常用）—— 正面影响

操作 有索引 无索引
SELECT 极大提升（O(log n)） 全表扫（O(n)），数据多时灾难
WHERE / JOIN 快速定位 笛卡尔积式扫表
ORDER BY 按索引顺序，避免 filesort 额外排序，额外内存/磁盘
GROUP BY 利用索引有序，避免临时表 Using temporary，性能极差
覆盖索引 不回表，减少 IO 必须回表

二、索引的代价 —— 负面影响

影响 说明
写入变慢 INSERT/UPDATE/DELETE 每次都要维护 B+ 树（分裂/合并/调整），索引越多写入越慢
占空间 一张表 5 个索引，索引空间可能超过数据本身
优化器选错 索引太多 → 优化器可能选错索引 → 反而变慢
锁范围扩大 UPDATE WHERE 如果没命中索引 → 行锁退化为表锁 → 并发度骤降

三、不同索引类型对性能的影响

索引类型 适用查询 性能影响 注意
主键索引 主键查询 最快（聚簇，不回表） 自增最好，UUID 导致页分裂频繁
唯一索引 等值 + 唯一校验 快（= 主键索引） 多了个唯一性检查，INSERT 比普通索引略慢
普通索引 等值 + 范围 好（回表一次） 覆盖索引能避免回表
联合索引 多条件查询 非常重要（减少索引数量） 必须遵循最左前缀
全文索引 MATCH...AGAINST 文本搜索极快 对 INSERT/UPDATE 开销很大
前缀索引 长字符串的等值查询 省空间 不支持 ORDER BY，区分度下降

11. Redis 的 RDB 和 AOF 机制

一句话：RDB 是定时快照，AOF 是写命令日志，两者互补实现 Redis 持久化。

一、RDB（Redis Database）—— 快照

每隔一段时间把内存中的全部数据生成一个 .rdb 二进制文件。

流程：fork() 子进程 → 子进程将当前数据写入临时文件 → 写完替换旧 RDB 文件 → 主进程不阻塞。

优点 缺点
✅ 二进制文件小，恢复快 ❌ 两次快照之间宕机 → 丢失这段时间的数据
✅ fork 子进程，不影响主进程 ❌ 数据量大时 fork 耗时，可能短暂阻塞
✅ 适合冷备、全量迁移 ❌ 不适合实时持久化

二、AOF（Append Only File）—— 写命令日志

把每条写命令追加到 .aof 文件末尾。

优点 缺点
✅ 最多丢 1 秒数据（everysec） ❌ 文件比 RDB 大得多（文本命令）
✅ 可读文本，误操作可手动改 ❌ 恢复速度比 RDB 慢（逐条重放）
✅ 适合实时持久化 ❌ 运行时间越长文件越大

三、AOF 重写（Rewrite）

AOF 文件越来越大 → 触发重写 → 用最精简的命令重建 AOF 文件。

重写流程：fork() 子进程 → 子进程按当前数据生成最小命令集写新文件 → 期间新增写操作同时写到旧 AOF + 重写缓冲区 → 子进程完成 → 重写缓冲区追加到新文件 → 新文件覆盖旧文件。

四、RDB vs AOF 对比

RDB AOF
文件大小 小（二进制压缩） 大（文本命令）
恢复速度 快 慢（逐条重放）
数据安全 可能丢失几分钟数据 最多丢 1 秒（everysec）
性能影响 fork 时短暂阻塞 持续 IO 写，everysec 影响小
文件可读 二进制不可读 文本可读
适用场景 冷备、迁移、快速恢复 实时持久化

12. Redis 过期键的删除策略

一、三种策略（Redis 采用：惰性 + 定期）

策略 做法 Redis 用了吗
定时删除 给每个过期 key 设一个定时器，到点就删 ❌ 没用（CPU 开销太大，key 多了定时器爆炸）
惰性删除 用到 key 时先检查，过期了就删 ✅ 用了
定期删除 每隔一段时间随机抽一批 key，删掉过期的 ✅ 用了
内存淘汰 内存满了，按规则淘汰数据 ✅ 兜底机制

二、惰性删除 —— 访问时检查

优点 缺点
CPU 友好（不主动找，用到了才查） 不访问就一直不删，内存浪费

三、定期删除 —— 定期随机抽查

优点 缺点
弥补惰性删除（清理不活跃的过期 key） 随机抽查可能漏掉

四、内存淘汰策略（兜底机制）

当内存满了，又没命中过期删除 → 按淘汰策略踢数据：

策略 行为
noeviction 不淘汰，写请求直接报错（默认 2.x，3.0+ 改为不淘汰只读）
allkeys-lru 所有 key 中，淘汰最久没用的（最常用）
volatile-lru 带过期时间的 key 中，淘汰最久没用的
allkeys-lfu 所有 key 中，淘汰使用频率最低的（Redis 4.0+）
volatile-lfu 带过期时间的 key 中，淘汰频率最低的
allkeys-random 所有 key 中随机淘汰
volatile-random 带过期时间的 key 中随机淘汰
volatile-ttl 带过期时间的 key 中，淘汰剩余过期时间最短的

五、总结：三层防线

问题 答案
Redis 过期删除用了几种策略？ 惰性删除 + 定期删除，兜底用内存淘汰
为什么不用定时删除？ key 多的时候定时器爆炸，CPU 扛不住
过期 key 一直没人访问会怎样？ 惰性删不了 → 依赖定期删除运气抽到 → 内存满了淘汰
缓存淘汰 LRU 和 LFU 区别？ LRU 看最近有没有用，LFU 看用得多不多；LFU 更适合热点不均匀的场景

13. Redis 单线程模型，为什么快？

Redis 的单线程：网络 IO 和命令执行都在一个主线程里完成（6.0+ 网络 IO 可多线程，命令执行仍是单线程）。

为什么单线程还这么快？

原因 说明
1. 纯内存操作 数据全在内存，读写纳秒级
2. 单线程，无锁竞争 没有上下文切换、没有死锁、没有锁开销
3. IO 多路复用 一个线程同时监听多个 socket，epoll 机制，有请求就处理，没请求就等着
4. 高效数据结构 SDS 简单动态字符串、ziplist、skiplist，专为 Redis 场景优化

Redis 6.0 的多线程跟这个矛盾吗？
不矛盾：6.0 的多线程只处理网络 IO 的读写（socket 数据的收发），命令执行仍然是单线程，数据竞争仍然不存在。

14. 缓存三大问题

问题 现象 解决方案
缓存穿透 查不存在的 key → 每次都打到 DB ① 布隆过滤器（不存在直接拦截）② 缓存空值（null，短 TTL）
缓存击穿 热点 key 过期 → 瞬间大量请求打到 DB ① 互斥锁更新（一个线程查 DB 重建，其他等）② 逻辑过期（永不过期，后台异步刷新）
缓存雪崩 大量 key 同时过期或 Redis 宕机 → DB 被压垮 ① TTL 加随机值 ② 多级缓存 ③ 主从 + 哨兵/集群保证高可用 ④ 限流熔断

15. Redis 事务

Redis 事务跟 MySQL 的 ACID 不同，它只保证一次性、顺序性、排他性执行。

特性 说明
不支持回滚 事务中某条命令执行失败，其他命令照常执行，不回滚
没有隔离级别 事务中的命令按序执行，执行期间其他客户端的命令不会插入

为什么 Redis 事务不支持回滚？ Redis 作者认为：语法错误应该在开发阶段发现；运行时错误（如 key 类型不匹配）是 bug，不应靠回滚掩盖。而且 Redis 追求简单和快，不支持回滚让实现更简洁。

Lua 脚本替代事务：原子性更强，一次发送一个脚本，整个脚本原子执行。

16. 分布式 ID 生成方案

方案 原理 优点 缺点
UUID UUID.randomUUID() 简单，无中心节点 无序 → B+ 树索引不友好；太长的字符串占空间
数据库自增 AUTO_INCREMENT 简单，有序 单点瓶颈；不适合分布式
号段模式 每个服务从 DB 一次拿一批 ID（如 1-1000），用完再取 减少 DB 访问 DB 仍是中心节点
Redis 自增 INCR id:order 原子自增 高性能 依赖 Redis 可用性
雪花算法（Snowflake）⭐ 64 位 long：1(符号位)+41(时间戳)+10(机器ID)+12(序列号) 高性能、有序、分布式 时钟回拨需要处理

处理时钟回拨：① 等待时钟追上 ② 换一个 workId ③ 用上一次的时间戳（全局停发）。

17. 分布式锁解决方案

方案 实现 可靠性
Redis（最常用） SET lock_key unique_value NX EX 30，释放用 Lua 脚本比 value 再删 高（单实例），更高（RedLock）
ZooKeeper 创建临时顺序节点，最小序号获锁，Watch 前一个节点 高（CP 强一致）
数据库 INSERT INTO 唯一键，用完删；或用 SELECT ... FOR UPDATE 低（性能差，基本不用）

Redis 分布式锁的关键：

  • 必须设过期时间（防止死锁）
  • 解锁必须用 Lua 脚本（判断 value 是自己的才删，保证原子性）
  • 主从架构下加锁可能丢失 → RedLock（向多个 Redis 实例分别加锁，多数成功才算）

18. 分布式事务解决方案

方案 原理 适用场景
2PC（两阶段提交） 协调者问所有人”准备好了吗？”→ 都回答好了 → 统一提交 单块跨库事务（性能差，XA）
TCC（Try-Confirm-Cancel） Try（预留资源）→ Confirm（确认执行）/ Cancel（释放资源） 资金转账类强一致性场景
本地消息表 本地事务 + 消息表一起写，定时任务扫未成功消息重试 订单-库存等最终一致性
RocketMQ 事务消息 发送半消息 → 执行本地事务 → 回查 → 投递/回滚 下单-发券等异步场景
Seata（AT 模式） 自动记录 undolog，一阶段提交 + 二阶段异步回滚 无侵入分布式事务

最常用：本地消息表（可靠消息最终一致性）+ RocketMQ 事务消息。

问题 答案
Redis 为什么单线程还快？ 纯内存 + 无锁 + IO 多路复用
缓存穿透怎么解决？ 布隆过滤器 + 缓存空值
雪花算法核心？ 时间戳(41位) + 机器 ID(10位) + 序列号(12位) = 64 位自增 ID
Redis 分布式锁怎么防误删？ Lua 脚本判断 value 再删，保证原子性
分布式事务最常用方案？ 本地消息表 + 定时重试（最终一致性）
