1. ConcurrentHashMap 原理及 JDK 1.7 与 1.8 的区别

ConcurrentHashMap 是线程安全的 HashMap，性能远优于 Hashtable。

JDK 1.7：分段锁（Segment + ReentrantLock）

将整个数组分成多个 Segment，每个 Segment 独立加锁，默认 16 个 Segment，最多支持 16 个线程同时写。

  • put：先 hash 定位 Segment → tryLock，失败则自旋 → 阻塞加锁 → 插入
  • get：不需要加锁（HashEntry 的 value 和 next 都是 volatile）

JDK 1.8：CAS + synchronized + 红黑树

彻底抛弃 Segment，改用 CAS + synchronized，结构与 HashMap 1.8 对齐。

put 流程（JDK 1.8）：

  1. key/value 都不能为 null
  2. table 为空 → CAS 初始化（延迟初始化）
  3. 桶为空 → CAS 直接插入，无需加锁
  4. 桶正在扩容（hash == MOVED）→ 当前线程帮忙扩容（helpTransfer）
  5. 桶非空 → synchronized 锁住头节点，遍历链表或树插入

get 流程：全程无锁，依赖 Node.val 和 Node.next 的 volatile 保证可见性。

JDK 1.7 vs 1.8 核心区别

维度 JDK 1.7 JDK 1.8
并发控制 Segment + ReentrantLock CAS + synchronized
数据结构 Segment 数组 + HashEntry 链表 Node 数组 + 链表 + 红黑树
锁粒度 段级别 桶级别（单个头节点）
查询复杂度 O(n/k) O(log n) 或 O(n)
初始化 创建时初始化 Segment 首次 put 才初始化（懒加载）
扩容 Segment 内独立扩容 多线程协同扩容
size 计算 三次不锁重试 + 全局加锁 baseCount + CounterCell（LongAdder 思想）

为什么比 Hashtable 快？

对比 Hashtable ConcurrentHashMap (1.8)
锁粒度 整个表一把锁 桶级别锁
读操作 加锁 无锁（volatile）
扩容 单线程 多线程协同

JDK 1.7 用分段锁降低锁竞争；JDK 1.8 用 CAS + synchronized 把锁粒度降到桶级别，加上红黑树和多线程协同扩容，并发性能大幅提升。

3. 什么是字节码？采用字节码有什么好处？

字节码（Bytecode）是 Java 源码经过 javac 编译后的中间表示（.class 文件），面向 JVM 而非 CPU，是 Java “一次编译，到处运行” 的基石。

编译与执行流程

用 javap -c 查看字节码：

每条指令一个字节（opcode），共约 200+ 条，基于操作数栈执行。

执行方式

方式 说明
解释执行 逐条翻译为机器码，启动快，运行慢
JIT 编译 热点代码编译为机器码缓存，运行快
混合模式（默认） 先解释，热点触发 JIT

五大好处

好处 说明
跨平台 字节码与 OS 无关，不同平台的 JVM 各自翻译为本地机器码
语言无关 Kotlin、Scala、Groovy 等都编译为 .class，跑在 JVM 上
动态增强 运行时生成/修改字节码（Spring AOP、MyBatis 的动态代理就是基于此）
安全可验证 加载时验证魔数、类型安全、栈溢出等，恶意字节码被拒之门外
性能优化 编译期做常量折叠 + 运行期 JIT 做方法内联、逃逸分析

字节码 vs 机器码

字节码 机器码
平台无关 ✅ 是 ❌ 否
执行速度 中等（JIT 后可接近） 最快
安全性 ✅ 可验证 ❌ 无

4. Java 类加载器

ClassLoader 负责将 .class 字节码加载到 JVM 内存为 Class 对象。

双亲委派模型

loadClass 流程：

为什么要双亲委派？

  • 安全：防止核心类被篡改（用户自定义 java.lang.String 不会生效）
  • 唯一：类的唯一标识 = ClassLoader + 全限定类名，避免重复加载

破坏双亲委派的场景

场景 为什么
JDBC DriverManager 由 Bootstrap 加载，但需要加载用户 classpath 下的 MySQL 驱动 → 用 Thread.currentThread().getContextClassLoader()
Tomcat 每个应用独立的 WebAppClassLoader，优先自己加载，保证不同应用的 Spring 版本不冲突
热部署 新建 ClassLoader 加载新版本，旧的全部 GC

类的完整加载过程

Class.forName() 会执行初始化，ClassLoader.loadClass() 只加载不初始化。

面试要点 答案
双亲委派怎么工作？ 先委托父加载器，父找不到再自己加载，递归到 Bootstrap
不同 ClassLoader 加载同一个类是同一个类吗？ 不是，类唯一标识 = ClassLoader + 全限定类名
Bootstrap 为什么打印 null？ C++ 实现，不是 Java 对象
