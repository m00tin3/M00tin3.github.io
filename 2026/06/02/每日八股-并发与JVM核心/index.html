1. ConcurrentHashMap 原理及 JDK 1.7 vs 1.8

JDK 1.7：Segment 分段锁

  • 默认 16 个 Segment，每个 Segment 继承 ReentrantLock + 内含 HashEntry 数组
  • put 只锁当前 Segment，get 无锁（value、next 都是 volatile）
  • Segment 数量初始化后不可变，扩容只扩 Segment 内部的 HashEntry
  • size 计算：先不加锁算 3 次，不一致再全局加锁

JDK 1.8：CAS + synchronized + 红黑树

  • 抛弃 Segment，结构与 HashMap 1.8 对齐（Node 数组 + 链表 + 红黑树）
  • 桶为空 → CAS 插入；桶非空 → synchronized 锁头节点
  • 正在扩容的桶（hash == MOVED）→ 当前线程帮忙扩容（helpTransfer）
  • size 用 baseCount + CounterCell 数组（LongAdder 思想，无锁分片计数）
维度 JDK 1.7 JDK 1.8
并发控制 Segment + ReentrantLock CAS + synchronized
锁粒度 段级别 桶级别（头节点）
数据结构 Segment + HashEntry Node + 链表 + 红黑树
查询复杂度 O(n/k) O(log n) 或 O(n)
初始化 创建时初始化 首次 put 延迟初始化
扩容 Segment 内独立 多线程协同（helpTransfer）
size 3次重试 + 全局锁 baseCount + CounterCell 无锁
key/value null 不允许 不允许（HashMap 允许一个 null key）

vs Hashtable：Hashtable 整个表一把锁（synchronized 方法），读也要加锁；CHM 桶级锁 + 无锁读 + 协同扩容，并发性能大幅提升。

JDK 1.7 分段降低锁竞争 → JDK 1.8 锁粒度降到桶级 + 红黑树 + 协同扩容，性能飞跃。

3. 什么是字节码？有什么好处？

字节码（.class）是 Java 源码经 javac 编译后的中间表示，面向 JVM 而非 CPU。每条指令一个字节（opcode），基于操作数栈执行。

执行方式：先解释执行（启动快），热点代码触发 JIT 编译为机器码（运行快），默认混合模式。

五大好处：

  1. 跨平台：字节码与 OS 无关，各平台 JVM 各自翻译
  2. 语言无关：Kotlin、Scala 等都编译为 .class，跑在 JVM 上
  3. 动态增强：运行时生成/修改字节码（Spring AOP CGLIB 代理、MyBatis Mapper 实现类）
  4. 安全验证：加载时验证魔数 CAFEBABE、类型安全、栈越界等
  5. 性能优化：编译期常量折叠 + 运行期 JIT 方法内联/逃逸分析

5. GC 如何判断对象可以被回收？

两种算法：引用计数法 和 可达性分析。

引用计数法（Java 没用）

每个对象维护一个引用计数器，被引用 +1，引用失效 -1，计数为 0 时回收。但无法解决循环引用：

Python、早期的 COM 用此法，Java 不采用。

可达性分析（Java 采用）

从一组 GC Roots 出发，沿着引用链向下搜索，走过的路径叫引用链。不在任何引用链上的对象 → 判定为不可达 → 可回收。

可作为 GC Roots 的对象 说明
虚拟机栈中引用的对象 方法局部变量、参数引用的对象
静态变量引用的对象 方法区中 static 引用指向的对象
常量引用的对象 字符串常量池、static final 引用指向的对象
JNI 引用 Native 方法引用的全局对象
活跃线程 运行中的线程对象及其 ThreadLocal
同步锁持有的对象 synchronized 持有的对象

判断过程中的两次标记

不是”不可达就立刻回收”，有两次标记：

一个对象的 finalize() 只会被调用一次，且 finalize() 已废弃（JDK 9 @Deprecated），因为它执行时机不确定、性能差、可用 try-finally 或 Cleaner 替代。

四种引用类型

引用类型 回收时机 用途
强引用 永不回收（除非不可达） Object obj = new Object()
软引用 SoftReference 内存不足时回收 缓存、图片缓存
弱引用 WeakReference GC 时必定回收 WeakHashMap、ThreadLocal 的 key
虚引用 PhantomReference 任何时候都可能 对象回收时的通知（ReferenceQueue）

小结

问题 答案
Java 用哪种算法？ 可达性分析，不用引用计数（循环引用问题）
从哪开始扫描？ GC Roots（栈、静态变量、常量池、活跃线程）
不可达就立即回收吗？ 不是，先标记 → finalize 自救机会 → 二次标记
软引用 vs 弱引用？ 软：内存不足才回收（适合缓存）；弱：GC 必回收（ThreadLocal key）
finalize() 还推荐用吗？ 不推荐，JDK 9 已废弃，用 Cleaner 或 try-finally

8. 线程安全 —— 从栈和堆的角度理解

一、为什么会有线程安全问题？一句话：多个线程共享了同一块堆内存。

JVM 内存模型决定了什么数据是线程安全的，什么不是：

内存区域 线程间共享？ 存放内容 线程安全？
栈（虚拟机栈） ❌ 每个线程独享 局部变量、方法参数、返回值地址 ✅ 天然安全
堆 ✅ 所有线程共享 对象实例、数组、成员变量 ❌ 需要额外手段保证
方法区 ✅ 共享 类信息、static 变量、常量池 ❌ static 变量需要额外保证

四、如何从堆的不安全变成安全？

本质就是把共享的堆数据变成**”串行访问”或“好像每个线程有自己的副本”**：

方式 原理 堆还是栈？
synchronized 锁住代码块，同一时刻只有一个线程执行 堆（锁住的是堆上的对象头）
volatile 保证可见性 + 禁止指令重排，但不保证原子性 堆（修饰的是堆中的变量）
ThreadLocal 每个线程在自己的堆里存一份副本 每个线程的 ThreadLocalMap 在堆里（但通过 key 隔离）
AtomicInteger（CAS） 无锁原子操作，CPU 指令级保证 堆（Unsafe 直接操作堆内存）
不可变对象 创建后不能改，共享也无所谓 堆（String、Integer 等本身就是堆对象）

五、一段代码说清楚栈安全、堆不安全

核心结论：局部变量在栈上 → 线程私有 → 天生安全；对象成员变量在堆上 → 线程共享 → 必须用同步手段保护。理解了这个，就知道为什么 Spring 的 Service 不要持有有状态的成员变量（否则高并发下数据错乱）。

问题 答案
为什么局部变量线程安全？ 在栈上，每个线程独享自己的栈帧，别的线程看不见
静态变量存在哪，安全吗？ 方法区（堆的延伸），所有线程共享，不安全
String 为什么安全？ 不可变，放在堆上也没人改得了
synchronized 锁在对象的哪？ 对象的对象头（Mark Word），在堆上
Spring Controller 为什么不能有成员变量？ Controller 是单例，成员变量在堆上被所有请求线程共享 → 数据错乱

9. 守护线程（Daemon Thread）

一句话：守护线程是后台服务线程，当所有非守护线程结束时，JVM 自动退出，不管守护线程有没有执行完。

特征

特征 说明
生命周期 依附于非守护线程，没有非守护线程了 → JVM 退出 → 守护线程强制结束
用途 后台辅助任务：GC、日志、定时任务、心跳检测
设置 thread.setDaemon(true)，必须在 start() 之前设置，否则抛异常
子线程 守护线程 new 出来的子线程默认也是守护线程
finally 所有非守护线程结束后，守护线程的 finally 不一定执行

运行结果：主线程睡了 3 秒后结束 → JVM 退出 → 守护线程的 while(true) 被强制终止。

为什么 finally 不一定执行？

JVM 退出时不会等守护线程优雅结束，而是直接终止所有守护线程，所以 finally 不可靠。

常见守护线程的例子

线程 说明
GC 线程 JVM 的垃圾回收就是守护线程
Tomcat 的 Acceptor 后台监听请求，主线程结束就停
定时任务调度 后台轮询任务
日志缓冲刷盘 定时把日志缓冲区写磁盘

非守护线程（用户线程） vs 守护线程

非守护线程（默认） 守护线程
创建 new Thread() 默认就是 setDaemon(true)
生命周期 独立，不依赖其他线程 依附于非守护线程
JVM 退出 有任何一个还在跑 → JVM 不退出 仅剩守护线程 → JVM 直接退出
finally ✅ 正常执行 ⚠️ 不一定执行
异常 未捕获异常只会结束当前线程 同上，但子守护线程也会被顺便干掉
问题 答案
守护线程和非守护线程的本质区别？ 非守护线程存活 → JVM 不退出；只剩守护线程 → JVM 直接退出
setDaemon(true) 什么时候调用？ 必须在 start() 之前，否则抛 IllegalThreadStateException
守护线程适合干什么？ 后台辅助任务：GC、监控、日志、定时清理
守护线程的 finally 可靠吗？ 不可靠，JVM 退出时不保证 finally 执行

10. ThreadLocal 的底层原理

一句话：每个线程内部有一个 ThreadLocalMap，ThreadLocal 作为 key，值是线程自己的副本，线程间互不干扰。

一、核心数据结构

每个 Thread 对象持有 threadLocals（ThreadLocalMap），ThreadLocalMap 内部是一个 Entry[] 数组，Entry 以 ThreadLocal 为 key。不同线程用同一个 ThreadLocal 对象访问时，实际是去各自线程的 Map 里取值。

三、Entry 的弱引用设计 —— 防止内存泄漏

引用类型 说明
Entry key（ThreadLocal） 弱引用 ThreadLocal 对象在外部没有强引用时，GC 时 key 被回收 → null
Entry value 强引用 即使 key 被回收了，value 仍然被 Entry 强引用，不会被 GC

四、内存泄漏问题及解决

如何解决？

方式 说明
**用完调用 remove()**（必须！） threadLocal.remove() → 删除整个 Entry，key 和 value 都能被 GC
ThreadLocal 自清理 get() 和 set() 时会顺便清理 key 为 null 的过期 Entry（expungeStaleEntry）
线程结束 线程终止 → ThreadLocalMap 整体被 GC → 没泄漏问题

线程池场景尤其重要：线程复用，ThreadLocalMap 一直存在，不 remove 的话 value 永远不被回收。

五、ThreadLocal 的应用场景

场景 说明

问题 答案
ThreadLocal 实现线程隔离的原理？ 每个 Thread 有自己的 ThreadLocalMap，ThreadLocal 为 key 操作各自的 Map
为什么 Entry 的 key 用弱引用？ 防止 ThreadLocal 对象自身无法被 GC，但弱引用只解决一半（value 还在）
内存泄漏怎么产生的？ key 弱引用被 GC → key=null，value 强引用还在 → 线程长期存活 → 泄漏
怎么避免泄漏？ **用完必须 remove()**，线程池场景尤其重要
ThreadLocalMap 怎么解决哈希冲突？ 线性探测法（不是链表），找下一个空位

11. ThreadLocal 内存泄漏问题及如何避免

一、泄漏是怎么发生的？三步走

第一步：ThreadLocal 对象外部强引用断开。

第二步：GC 触发，Entry 的 key（弱引用）被回收。

第三步：线程长期存活（线程池），泄漏不断累积。

二、为什么不用强引用？为什么弱引用只解决一半？

如果 key 是… 后果
强引用 ThreadLocal 对象外部引用断了，但 Entry 强引用还指着它 → ThreadLocal 自己也 GC 不掉 → 泄漏更严重
弱引用（JDK 的选择） ThreadLocal 可以被 GC，但 value 还是 GC 不掉 → 只解决了”一半”

JDK 选择弱引用是为了让 ThreadLocal 对象本身能被 GC，但这只解决了 key 的问题，value 依然泄漏。所以 JDK 还加了自清理机制。

三、JDK 的自清理机制 —— 被动防御

问题 答案
泄漏根因是什么？ Entry 的 value 是强引用，ThreadLocal 被 GC 后 value 仍然可达
为什么线程池更严重？ 线程复用 → ThreadLocalMap 长期存活 → 过期 value 越积越多
自清理能完全避免泄漏吗？ 不能，只在你后续用 get/set 时才清理，不用就不触发
最可靠的解决方案？ finally { threadLocal.remove(); }，无例外
怎么排查 ThreadLocal 泄漏？ 堆 dump → 查 Thread 的 threadLocals 字段 → 找 key=null 的大 Entry

12. 并行、串行、并发

一、一句话区分

串行（Serial） 并发（Concurrent） 并行（Parallel）
核心 一个任务做完再做下一个 交替执行，看起来像同时 真正的同时执行
CPU 单核就够 单核即可 必须多核
比喻 一个人在流水线做完所有工序 一个人带三个娃，来回切换哄 三个人各自带一个娃

二、图解理解

三、并发 vs 并行 —— 核心区别

对比 并发 并行
物理条件 单核或多核都可以 必须有多核 CPU
同时性 逻辑上同时（交替快得像同时） 物理上真正同时
关注点 任务的调度和切换 任务的同时执行
问题 竞态条件、死锁、上下文切换开销 也要处理竞态条件，但多了 CPU 缓存一致性等问题
Java 实现 synchronized、线程池、CompletableFuture ForkJoinPool、parallelStream()

四、为什么需要区分？—— 线程数该怎么设？

任务类型 线程数公式 原因
CPU 密集型 核心数 + 1 并行是关键，线程多了切换开销反而大
IO 密集型 核心数 * 2 或 核心数 / (1 - 阻塞系数) 并发是关键，线程多数在等待 IO，多开几个让 CPU 不闲着

五、Java 中的体现

技术 属于 说明
Thread / Runnable 并发 单核也可，靠时间片轮转
synchronized / Lock 并发控制 解决并发下的竞态条件
ForkJoinPool 并行 将大任务拆分，多核并行执行
parallelStream() 并行 数据分片，每个核心处理一部分
CompletableFuture 并发 异步编排，不阻塞主线程
问题 答案
并发一定要多核吗？ 不需要，单核靠时间片轮转即可实现并发
并行一定需要多核吗？ 必须，没有多核谈不上”同时执行”
线程多就一定快吗？ 不一定，过度并行导致上下文切换开销反而变慢
parallelStream() 适合什么场景？ 大数据量、独立计算、CPU 密集型；小数据量用串行更快
