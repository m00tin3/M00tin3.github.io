1. 并发三大特性：原子性、可见性、有序性

一句话：并发编程的所有 bug 都能归结为这三大问题 —— 原子性、可见性、有序性被打破。

特性 问题 原因 解决方案
原子性 一个操作被打断 CPU 时间片轮转，线程切换 synchronized、Lock、AtomicInteger（CAS）
可见性 一个线程改了，另一个看不到 CPU 缓存 + JMM 工作内存 volatile、synchronized、Lock
有序性 代码执行顺序被打乱 编译器/CPU 指令重排 volatile、synchronized、Lock

一、原子性

一个或多个操作要么全部执行完且不被打断，要么全不执行。

方式 说明
synchronized 锁住代码块，串行执行
Lock 显式锁，粒度更灵活
AtomicInteger / CAS CPU 指令级原子操作，无锁

二、可见性

一个线程修改了共享变量，其他线程能立刻看到修改后的值。

问题出在 JMM（Java 内存模型）：

可见性的两个来源：

  1. CPU 缓存（L1/L2/L3）：每个核心有自己的缓存，修改后不一定立刻写到主存
  2. JMM 工作内存：Java 抽象层面，每个线程有自己的工作内存（类比 CPU 缓存）
方式 如何保证可见性
volatile 写之后立即刷回主内存，读之前从主内存拿最新值（内存屏障）
synchronized JMM 规定：解锁前必须把工作内存的修改刷回主内存
Lock 同 synchronized，释放锁前刷回

三、有序性

代码的执行顺序可能和你写的不一样，有两个层面：

  1. 编译器重排：编译器优化，调整指令顺序（不破坏单线程语义）
  2. CPU 指令重排：CPU 流水线优化，乱序执行

volatile 防重排原理：在指令间插入内存屏障（Memory Barrier），禁止屏障前后的指令交换顺序。

volatile vs synchronized

volatile synchronized
原子性 ❌ 不保证（i++ 照样不安全） ✅ 保证
可见性 ✅ 保证 ✅ 保证
有序性 ✅ 保证（禁止指令重排） ✅ 保证
性能 高（无锁） 低（有锁竞争）
适用场景 标志位、状态标记、DCL 单例 需要原子操作的代码块

三大特性总结

问题 答案
并发三大特性是什么？ 原子性、可见性、有序性
i++ 写了一个语句，安全吗？ 不安全，i++ 不是原子操作（读→改→写），用 AtomicInteger 或 synchronized
volatile 能替代 synchronized 吗？ 不能，volatile 不保证原子性
DCL 单例为什么必须 volatile？ 禁止 new Singleton() 的指令重排，防止返回半成品对象
synchronized 怎么保证可见性？ JMM 规定：解锁前刷回主内存，加锁后从主内存读最新值

2. 线程池为什么要用阻塞队列？为什么先加队列而不是直接创建到最大线程？

一、为什么是阻塞队列而不是普通队列？

用普通队列 用阻塞队列
队列满 → add() 抛异常 / offer() 返回 false 队列满 → put() 阻塞，等有空位
消费者线程需要轮询队列（while 循环一直问有没有新任务） 消费者自动被唤醒
空转浪费 CPU 线程挂起，不占 CPU

阻塞队列的核心能力：队列空时，取元素的线程阻塞等待；队列满时，放入元素的线程阻塞等待。 线程池的工作线程最终调用 queue.take()，队列空时自动阻塞，不用轮询。

二、线程池的执行顺序：核心线程 → 队列 → 最大线程 → 拒绝

三、为什么先加队列，而不是先创建到最大线程？

两个原因：性能和缓冲。

原因一：避免频繁创建/销毁线程

原因二：队列是缓冲区，避免线程爆炸

四、一句话总结这个设计哲学

核心线程 = 日常处理能力；队列 = 缓冲瞬时流量；最大线程 = 临时救急资源；拒绝策略 = 最后的兜底。优先级：复用 > 缓冲 > 应急 > 拒绝。

五、补充：这个设计也带来一个”坑”

当核心线程满了，任务在队列中排队，即使 maxPoolSize 还没满，也不会创建新线程。只有队列也满了，才会扩线程。所以如果你的有界队列设得太大（比如 1000），可能永远扩不到 maxPoolSize——队列还没满就够装了，临时线程永远不会被创建。

问题 答案
为什么一定要用阻塞队列？ 消费者 take() 阻塞等，不用轮询空转 CPU
先入队列而不是先扩线程的原因？ 线程复用优先，降低创建/销毁开销；队列缓冲瞬时流量，防止线程数爆炸
什么时候才会创建到 maxPoolSize？ 核心线程和队列都满了，才会扩线程
队列设太大有什么坑？ 可能永远扩不到 maxPoolSize，临时线程永不创建

3. 线程池的线程复用原理

一句话：线程池的线程不会执行完 run() 就死掉，而是进入一个死循环，不断从阻塞队列 take() 任务，队列空就阻塞等待，来任务就继续执行。

一、核心：Worker 线程的 runWorker()

二、getTask() —— 从阻塞队列拿任务

三、复用的本质 —— 一张图说清楚

普通线程是一次性的：start() → run() 执行完 → 销毁。
线程池 Worker 是循环的：run() 执行完 → 继续 take() → 拿到新任务 → 再执行 → 循环往复。

四、核心线程 vs 非核心线程的回收

线程类型 take/poll 行为
核心线程 workQueue.take() 无限阻塞 队列空时挂起等，不消耗 CPU，永远不会死
非核心线程 workQueue.poll(keepAliveTime) 超时 空闲超过 keepAliveTime 还没拿到任务 → getTask() 返回 null → 退出循环 → 线程销毁

五、为什么叫”复用”而不是”常驻”

问题 答案
Worker 线程怎么做到不死的？ while 循环 + queue.take() 阻塞等，不是执行完就 return
核心线程和非核心线程的本质区别？ 核心线程 take() 无限等，非核心线程 poll(keepAliveTime) 超时回收
线程复用的是什么东西？ OS 线程资源（不是 run 方法），同一个线程反复执行不同 task.run()
task.run() 为什么不调 start()？ start() 会创建新线程，破坏复用；run() 在当前 Worker 线程里直接执行
线程退出的唯一时机？ getTask() 返回 null → while 循环结束 → Worker 被销毁

4. Spring 是什么？

一句话：Spring 是一个轻量级的 Java 开发框架，核心是 IOC（控制反转） 和 AOP（面向切面编程），目的是简化 Java 企业级开发。

一、Spring 解决的核心问题

二、两大核心

核心 做什么 解决的问题
IOC（控制反转） 对象创建、管理、注入全交给 Spring 容器，不需要自己 new 解耦：Service 不依赖具体 Dao 实现，靠接口 + 注入
AOP（面向切面编程） 把横切逻辑（日志、事务、权限）从业务代码中抽出来 关注点分离：业务代码只写业务逻辑，切面干脏活

Spring Security 认证授权
Spring Cloud 微服务全家桶（注册中心、配置中心、网关）

二、详细步骤（共 8 步）

步骤 阶段 Spring 容器做了什么
1. BeanDefinition 注册 扫描阶段 @ComponentScan → 扫描 .class → 生成 BeanDefinition → 放入 beanDefinitionMap
2. BeanFactoryPostProcessor 扫描阶段 在所有 Bean 实例化之前，修改 BeanDefinition（如 ${} 占位符解析）
3. 实例化 createBean BeanFactory 反射调用构造函数或工厂方法 newInstance()，创建 Bean “空壳”
4. 属性填充 createBean 自动注入 @Autowired、@Value 字段，调用 setter
5. Aware 回调 createBean 如果实现了 BeanNameAware、BeanFactoryAware、ApplicationContextAware，依次回调
6. BeanPostProcessor 前置 initializeBean 所有 BeanPostProcessor 的 postProcessBeforeInitialization() 被调用
7. 初始化 initializeBean @PostConstruct → InitializingBean.afterPropertiesSet() → init-method
8. BeanPostProcessor 后置 initializeBean postProcessAfterInitialization()（AOP 代理在这里生成，可能返回代理对象替代原 Bean）
9. 就绪 放入 singletonObjects 一级缓存
10. 销毁 容器关闭 @PreDestroy → DisposableBean.destroy() → destroy-method

可做什么 修改 Bean 的类名、作用域、属性等元信息 修改/替换 Bean 实例（生成代理）

7. Spring Bean 的几种作用域

作用域 说明 生命周期
singleton（默认） 整个容器只有一个实例 容器启动创建 → 容器关闭销毁
prototype 每次 getBean() 都新建一个 创建后 Spring 不再管理，销毁自己负责
request 同一个 HTTP 请求内共享同一个 请求结束销毁（仅 Web 环境）
session 同一个 HTTP Session 内共享同一个 Session 过期销毁（仅 Web 环境）
application 整个 ServletContext 共享一个 Web 应用关闭销毁（仅 Web 环境）
websocket 同一个 WebSocket 会话内共享 WebSocket 关闭销毁

一、singleton（默认）—— 最常用

Spring 容器只维护一个实例，所有用到的地方共享同一个对象：

优点 缺点
性能好，无需反复创建 不能持有有状态字段，否则并发下数据错乱

为什么 Controller 和 Service 默认单例？因为它们是无状态的——只调方法，不存数据。所有线程共用一个实例，线程安全问题靠不存状态来规避。

二、prototype —— 每次新建

注意 说明
销毁回调不执行 Spring 创建完就不管了，@PreDestroy 不会触发
适合有状态场景 每个线程拿自己的实例，互不影响
懒加载 prototype Bean 是懒加载的，不像 singleton 在启动时创建

三、request / session —— Web 环境专用

使用场景：

  • request：请求上下文（当前请求的 traceId、用户信息），请求结束自动销毁
  • session：登录用户信息、购物车，Session 过期自动销毁

proxyMode 必须设置的原因：singleton Bean（如 Controller）注入 request 作用域的 Bean 时，Controller 只有一个，但 request Bean 每个请求都不同 → 需要代理，让 singleton Bean 每次调用都路由到当前请求的那个实例。

四、怎么选？

场景 选什么
无状态 Service、Controller（99% 场景） singleton
有状态对象，每次用新实例 prototype
请求级上下文信息（traceId、当前用户） request
用户会话数据（登录态、购物车） session
问题 答案
默认作用域是什么？ singleton，容器中只有一个实例
prototype Bean 的 @PreDestroy 会执行吗？ 不会，Spring 不管 prototype 的销毁
单例 Bean 注入原型 Bean 有什么问题？ 原型 Bean 只会被注入一次（单例依赖不变），需要 @Lookup 或 ObjectFactory 或 proxyMode 解决

结论：不保证。线程安全取决于 Bean 本身有没有状态，和 Spring 无关。

一、为什么说不保证？

Spring 单例 Bean = 容器中只有一个实例 = 多个线程共用同一个对象。

二、安全 vs 不安全的三类情况

情况 线程安全？ 说明
无状态的 Bean（不保存数据，只调方法） ✅ 安全 Controller、Service、Dao 通常都是无状态的
有不可变成员变量（final、String、注入的 Service） ✅ 安全 @Autowired 注入的成员在启动后就不变了
有读写型成员变量（用户数据、计数器等） ❌ 不安全 多线程同时写 → 竞态条件

三、如果确实需要有状态怎么办？

方案 做法 适用场景
不要用成员变量 用局部变量、方法参数 最佳实践
改用 prototype @Scope("prototype") 每次新实例
ThreadLocal 线程隔离，一人一份 请求上下文（当前用户、traceId）
加锁 synchronized 串行化访问 不推荐，性能差

四、Spring 管不管线程安全？

Spring 容器只管 Bean 的创建和注入，不负责线程安全。它不会对单例 Bean 加锁（那也太慢了）。保证线程安全是你写业务代码时自己要考虑的。

谁负责 做什么
Spring 容器 创建 Bean、注入依赖、管理作用域、生命周期
你自己 保证 Bean 是无状态的，或者用 ThreadLocal/prototype/锁保证有状态场景的安全

所以 Controller、Service、Dao 都应该设计成无状态的——只调方法，不存数据。这是 Spring 开发的第一条铁律。

问题 答案
Spring 单例 Bean 线程安全吗？ 不保证，取决于 Bean 本身是否无状态
为什么无状态就安全？ 没有共享的、可改的数据，各线程走各自的局部变量
注入的 Dao 算”有状态”吗？ 不算，注入的引用在启动时就固定了，之后不再变
有状态该用什么方案？ ThreadLocal > prototype > 加锁（性能递减）
为什么不给单例 Bean 自动加锁？ 加锁 → 所有请求串行化 → 性能灾难

9. Spring 用了哪几种设计模式？

设计模式 在 Spring 中的体现 一句话
工厂模式 BeanFactory、ApplicationContext 通过 getBean() 获取对象，不用自己 new

责任链模式 Filter 链、Interceptor 链 一个请求经过层层处理，每层干每层的事

Spring AOP 用了什么模式？ 代理模式（底层）+ 责任链模式（通知链）+ 策略模式（选 JDK 还是 CGLIB）
工厂模式在哪体现？ BeanFactory / ApplicationContext 通过 getBean() 创建对象
模板方法模式解决了什么问题？ 框架写死流程（连接获取→执行→释放），只暴露变化部分（SQL）给你
适配器模式在 Spring MVC 中的作用？ HandlerAdapter 适配不同类型的 Controller，让 DispatcherServlet 统一调用

10. Spring 事务的实现方式与隔离级别

一、两种实现方式

方式 做法 优缺点
声明式事务（@Transactional） 注解 + AOP，声明方法需要事务 ✅ 无侵入、简单；❌ 粒度是方法级
编程式事务（TransactionTemplate） 手动写代码控制事务边界 ✅ 粒度灵活；❌ 代码侵入

99% 的场景用 @Transactional，编程式只在需要精确控制事务边界时用。

二、@Transactional 的底层原理 —— AOP 代理

@Transactional 能生效，必须满足三个条件：

  1. 类被 Spring 管理（Bean）
  2. 方法被 public 修饰（AOP 代理只能拦截 public 方法）
  3. 外部调用（同一个类里 this.方法() 不会走代理，事务失效）

三、事务失效的常见场景

场景 为什么失效 怎么解决
同类调用 this.method() 不走代理 拆到不同类，或用 AopContext.currentProxy()
非 public 方法 AOP 不拦截 private/protected 改为 public
异常被吃了 try-catch 没抛出去，Spring 不知道要回滚 catch 里手动 TransactionAspectSupport.currentTransactionStatus().setRollbackOnly()
抛非运行时异常 默认只对 RuntimeException 和 Error 回滚 @Transactional(rollbackFor = Exception.class)

四、隔离级别

隔离级别 说明 解决的问题
DEFAULT（默认） 跟随数据库的隔离级别 —
READ_UNCOMMITTED 读未提交 什么都解决不了
READ_COMMITTED 读已提交 解决脏读
REPEATABLE_READ 可重复读 解决脏读 + 不可重复读（MySQL 默认）
SERIALIZABLE 串行化 解决脏读 + 不可重复读 + 幻读，性能最差

五、传播行为

传播行为 说明
REQUIRED（默认） 有事务就加入，没有就新建 → 99% 用这个
REQUIRES_NEW 不管有没有事务，都新开一个独立事务
SUPPORTS 有事务就加入，没有就非事务运行
NOT_SUPPORTED 挂起当前事务，非事务运行
MANDATORY 必须在事务中，否则抛异常
NEVER 必须在非事务中，否则抛异常
NESTED 嵌套事务（创建 savepoint）
传播行为 场景
REQUIRES_NEW 记日志（日志不能因为主业务回滚而回滚）
NESTED 批量处理（部分失败不影响整体）
问题 答案
@Transactional 底层怎么实现的？ AOP 代理：代理类在方法前后织入事务逻辑（开/提交/回滚）
为什么同类调用事务失效？ this 是原对象，不走代理 → AOP 拦截不到
默认什么异常会回滚？ RuntimeException 和 Error，受检异常不会
声明式和编程式怎么选？ 99% 用 @Transactional，精细控制时用 TransactionTemplate
MySQL 默认隔离级别？ REPEATABLE_READ（可重复读），Oracle 是 READ_COMMITTED

12. Bean 的自动装配及方式

自动装配：Spring 容器自动查找并注入 Bean 的依赖，不需要手动 new 和赋值。

一、三种注入方式

方式 示例 推荐度
字段注入 @Autowired private UserDao dao; ⭐⭐ IDEA 都警告
setter 注入 @Autowired public void setDao(UserDao dao) ⭐⭐⭐
构造器注入 public UserService(UserDao dao) ⭐⭐⭐⭐⭐ 官方推荐

为什么构造器注入最好？

对比 字段注入 构造器注入
可否用 final ❌ 不能 ✅ 可以 → 不可变，更安全
依赖是否必填 不清楚 ✅ 构造器参数明确是必填
NPE 风险 容易 NPE（字段可能没注入） 编译期保证，不可能 NPE
单元测试 需要反射注入 直接 new 传入 mock → 方便测试
循环依赖 能启动（隐藏问题） 直接报错 → 及早暴露

二、@Autowired 的匹配策略

三、@Resource vs @Autowired

@Autowired @Resource
来源 Spring JDK（javax.annotation）
匹配策略 默认按类型（byType） 默认按名称（byName）
按名称指定 @Qualifier("name") @Resource(name = "name")
使用 Spring 项目更常用 JDK 标准，解耦

四、自动装配的边界 case

场景 行为 解决
容器中没有匹配的 Bean 抛 NoSuchBeanDefinitionException @Autowired(required = false)
有多个匹配的 Bean 抛 NoUniqueBeanDefinitionException @Qualifier / @Primary
集合注入（List<UserDao>） 注入所有 UserDao 实现类 自动收集
Map 注入（Map<String, UserDao>） key=Bean 名，value=Bean 实例 自动收集
问题 答案
推荐哪种注入方式？ 构造器注入，Spring 官方推荐，可 final + 不 NPE + 测试方便
@Autowired 匹配规则？ 先按类型 → 多个按名称 → 找不到抛异常
@Autowired 和 @Resource 的区别？ @Autowired 默认 byType（Spring）；@Resource 默认 byName（JDK）
为什么字段注入被 IDEA 警告？ 不可 final、易 NPE、不好测试、隐藏循环依赖
