1. Spring、Spring Boot、Spring MVC 的区别

对比 Spring Spring MVC Spring Boot
定位 核心框架（基础底座） Web 层框架（处理 HTTP） 快速构建工具（开箱即用）
核心能力 IOC + AOP DispatcherServlet + Controller 自动配置 + starter + 内嵌服务器

  • Spring Boot = 脚手架，帮你自动配置好 Spring + Spring MVC 等一堆东西，一键启动

6. MyBatis 的优缺点

一、MyBatis 是什么

SQL 和 Java 代码分离的半自动 ORM 框架：SQL 写在 XML/注解里，框架负责把结果映射为 Java 对象，但 SQL 你来写。

二、优点

优点 说明
SQL 可控 自己写 SQL，复杂查询、多表关联、性能调优完全自主，不会被框架生成烂 SQL
动态 SQL <if>、<foreach>、<choose> 灵活拼 SQL，条件查询不用拼字符串
轻量 没有 Hibernate 的自动建表、脏检查、一级缓存那么重的机制，上手快
松耦合 SQL 独立在 XML 文件中，不改 Java 代码就能调 SQL
与 Spring 集成好 mybatis-spring-boot-starter 引入即用，@Mapper 接口代理

三、缺点

缺点 说明
SQL 都得自己写 简单的 CRUD 也要手写 SQL，不像 JPA 自动生成
数据库移植难 SQL 强绑定特定数据库语法（分页、函数），换数据库 = 改大量 SQL
复杂的映射关系配置多 一对多、多对多、嵌套查询要写 resultMap，层级深时很繁琐
无对象缓存 MyBatis 只有一级缓存（SqlSession 级别，很短），二级缓存弱，不像 Hibernate 有完善的多级缓存

四、MyBatis vs JPA/Hibernate

MyBatis JPA/Hibernate
定位 半自动（SQL 你写，框架只做映射） 全自动（CRUD SQL 自动生成，实体管理）
SQL 控制 ✅ 完全可控 ❌ 自动生成，复杂查询不可控
简单 CRUD 要手写 SQL 一个 save() 搞定
多表/复杂查询 ✅ 手写优化 ❌ 自动 SQL 可能烂
移植性 ❌ 数据库强绑定 ✅ 方言抽象，换库省事
学习成本 低（会写 SQL 就行） 高（实体状态、级联、懒加载）
适用场景 SQL 密集型、报表、复杂查询 标准 CRUD、快速开发

五、怎么选？

场景 选择
互联网项目，SQL 复杂，要调优 MyBatis
企业内部系统，表单增删改查为主 JPA
两者都要 MyBatis-Plus（简单 CRUD 自动生成 + 复杂查询手写 SQL）
问题 答案
MyBatis 是 ORM 吗？ 是半自动 ORM，SQL 和映射分离，自己写 SQL
MyBatis 缓存？ 一级缓存 SqlSession 级别，二级缓存 namespace 级别（默认关闭）
#{} 和 ${} 区别？ #{} 预编译占位符（防注入），${} 直接拼接（有注入风险，动态表名/列名时才用）
MyBatis-Plus 是什么？ MyBatis 增强工具，自动生成 CRUD、分页、逻辑删除，减少手写 SQL

7. SQL 注入

一句话：攻击者把恶意的 SQL 片段拼接到查询语句中，让数据库执行了不该执行的逻辑，导致数据泄露、篡改或删除。

一、怎么被注入的？—— 字符串拼接

二、预编译（PreparedStatement）为什么能防注入？

预编译的两步：

  1. 编译：把 SQL 发送给数据库 → 数据库解析 SQL 结构，确定要查什么
  2. 传参：把参数值发给数据库 → 数据库只把参数当数据，不会重新解析 SQL 结构

三、MyBatis 中 #{} vs ${}

#{} ${}
原理 预编译占位符（PreparedStatement 的 ?） 直接拼接字符串
是否防注入 ✅ 防注入 ❌ 不防
何时用 99% 的场景 动态表名、列名、ORDER BY（这些不能 ?）

四、白名单校验 ${} 参数

五、常见注入类型

类型 攻击方式 后果
始终真 ' OR 1=1 -- 查出所有数据
联合查询 ' UNION SELECT username, password FROM admin -- 偷敏感数据
注释符 admin'-- 绕过密码校验（SQL 注释掉密码部分）
堆叠查询 '; DROP TABLE t; -- 删表（部分数据库支持）
盲注 根据返回/延迟判断是否存在数据 没有回显时也能猜数据

六、防御手段

方式 说明
预编译 99% 场景用 PreparedStatement（MyBatis 的 #{}）
白名单 ${} 必须配白名单校验
ORM 框架 JPA、MyBatis 默认防注入（只要别用 ${}）
参数校验 对输入长度、类型做校验
最小权限 数据库用户只给需要的权限（不让删表）
问题 答案
SQL 注入的根因？ 用户输入当成 SQL 代码执行了
预编译为什么能防？ 编译结构先行，参数只当数据不参与解析
#{} 和 ${} 区别？ #{} 预编译防注入；${} 字符串拼接不防
${} 什么场景必须用？ 动态表名、列名、ORDER BY → 必须白名单校验
登录是否被注入怎么测？ 输入 ' OR '1'='1 / ' OR 1=1 --，正常返回就说明被注入了

9. 索引的基本原理

一句话：索引就像书的目录，帮你快速定位数据，避免全表扫描。数据库索引底层一般是 B+ 树。

一、没有索引 vs 有索引

无索引（全表扫描） 有索引（B+ 树）
查找方式 一行行扫 导航式查找
复杂度 O(n) O(log n)
2000 万行 一次查 2000 万行 约 3-4 次 IO

二、索引背后的数据结构：B+ 树

**

为什么是 B+ 树而不是二叉树/哈希表？

数据结构 问题
二叉树 数据递增时退化成链表（如 1→2→3→…→10000），查询变 O(n)
红黑树 虽然是平衡的，但只有两个分叉，高度大、IO 次数多
哈希表 只能精确匹配 =，无法范围查询 > BETWEEN，无法排序
B+ 树 ✅ 多叉（矮胖）→ 层数少 → IO 少；叶子有序链表 → 范围查询快

三、B+ 树的核心特性

特性 说明
多叉 一个节点可以存多个 key，叉数大 = 树矮 = IO 少（一个节点 = 一个磁盘页）
数据全在叶子节点 非叶子节点只存 key（不存数据）→ 一个节点能存更多 key → 树更矮
叶子节点有序 + 双向链表 范围查询 BETWEEN 100 AND 200 找到起点后顺着链表走就行，不用回树
自平衡 节点分裂/合并自动维护平衡，始终 O(log n)

四、InnoDB 中索引的具体实现

主键索引（聚簇索引）：叶子节点存的是整行数据。

普通索引（二级索引）：叶子节点存的是主键值，需要回表。

覆盖索引：如果查的字段全在索引里了，不用回表。

五、什么时候建索引？什么时候不建？

该建 不该建
WHERE 后面的条件列 很少用到的列
JOIN 的关联列 内容变化频繁的列（每改一次索引要重建）
ORDER BY 的排序列 数据很少的表（几百行，直接扫就完了）
区分度高的列（性别就不适合，只有男/女两种） 区分度低的列
问题 答案
索引底层是什么？ B+ 树（多叉平衡树）
为什么 B+ 树叶子节点连起来？ 范围查询顺着链表走，不用一次一次回树
回表是什么？ 普通索引拿到主键 → 主键索引再查一次才拿到完整行数据
覆盖索引是什么？ 查询字段全在索引里 → 不用回表 → Using index
一个表建索引越多越好吗？ 不是，索引也占空间，增删改都要维护索引，过多反而慢

10. MySQL 聚簇索引和非聚簇索引的区别

一、核心区别一句话

聚簇索引的叶子节点直接存整行数据，非聚簇索引的叶子节点存主键值（想拿完整数据必须回表）。

二、一张图看懂

三、详细对比

聚簇索引 非聚簇索引
别名 主键索引 二级索引 / 辅助索引
叶子节点存什么 整行数据 主键值（不是物理地址，而是主键）
一个表能有几个 一个（数据只能按一种顺序物理存储） 多个（可以有多个二级索引）
是否回表 查主键 → 直接拿到整行数据 查普通列 → 拿主键 → 回表找主键索引
主键是否自动创建 InnoDB 有主键就用，没主键用第一个唯一索引，都没有 自动建隐藏_rowid
物理存储 叶子节点即数据本身（索引和数据在一起） 索引和数据分离（叶子只有主键值）

四、InnoDB 和 MyISAM 的区别

InnoDB MyISAM
主键索引 聚簇索引（叶子存整行数据） 非聚簇（叶子存数据行的物理地址）
普通索引 叶子存主键值 + 回表 叶子存数据行的物理地址
数据文件 .ibd（索引即数据） .MYD（数据文件）+ .MYI（索引文件）分离
事务 ✅ 支持 ❌ 不支持

五、为什么 InnoDB 普通索引存主键而不是物理地址？

存物理地址 存主键
页分裂/数据移动时 → 所有索引都要更新地址 只需更新主键索引，普通索引不用动
维护成本极高 ✅ 维护成本低

六、为什么建议用自增主键？

问题 答案

回表是什么？ 二级索引拿到主键 → 主键索引再查整行 → 多一次 IO
自增主键 vs UUID 主键？ 自增写入快（不分裂）、索引瘦；UUID 页分裂多、索引胖
