1. JDK、JRE、JVM 是什么？有什么区别？

定义

名词 全称 是什么
JVM Java Virtual Machine Java 虚拟机，执行 .class 字节码，是 Java “一次编译，到处运行” 的核心
JRE Java Runtime Environment Java 运行时环境 = JVM + 核心类库（rt.jar等），只负责运行 Java 程序
JDK Java Development Kit Java 开发工具包 = JRE + 开发工具（javac、javap、javadoc、jdb等），包含编译和运行能力

三者关系（包含关系）

JDK ⊃ JRE ⊃ JVM

面试要点

问题 答案
只运行 Java 程序，需要什么？ 只需 JRE
要开发/编译 Java 源码，需要什么？ 需要 JDK（JDK 已包含 JRE）
Java 跨平台的原理靠谁？ JVM，每个平台有对应的 JVM 实现，屏蔽了底层操作系统差异
JVM 能用 Java 写吗？ 不能。JVM 本身是 C/C++ 实现的，部署在每个操作系统上

JDK 各版本关系

从 JDK 11 开始，Oracle 移除了独立的 JRE 发布，JDK 自带了运行时环境。通常使用 jlink 工具按需生成定制化的运行时镜像。

核心工具一览

工具 作用
javac 编译器，.java → .class
java 启动 JVM，运行字节码
javap 反编译 .class 文件，查看字节码
jps 查看当前系统上运行的 Java 进程
jstat JVM 统计信息监控
jstack 查看线程堆栈，排查死锁
jmap 生成堆转储（heap dump）
jconsole 图形化 JVM 监控工具

3. final 关键字的作用

final 可以修饰类、方法、变量（含参数），表示”不可改变”。

一张表总结

修饰目标 效果 举例
类 不能被继承 public final class String {}
方法 不能被子类重写（Override） public final void run() {}
基本类型变量 值不可变（常量） final int MAX = 100;
引用类型变量 引用不可变，但对象内容可改 final List<String> list = new ArrayList<>() → 可以 add
参数 方法内不能重新赋值 void foo(final int x) {}

代码详解

final 变量初始化（空白 final）

final 变量可以不声明时赋值，但必须在使用前且构造方法结束前完成初始化：

final 修饰局部变量与匿名内部类

final 与 static 组合

组合 含义
final 实例变量 每个对象一个，不可变
final static 类级别全局常量，只有一份
final static + 基本类型/String 编译期常量，编译器直接内联替换

final 的性能优势（JVM 层面）

  • final static 常量在编译期被直接内联到调用处，无需字段查找
  • final 方法可被 JIT 编译器做激进优化（因为不存在子类重写的不确定性）
  • final 类的方法默认都是 final 的

final 不能修饰构造方法

构造方法天生不能被继承/重写，所以 final 修饰构造方法没有意义，编译报错。

4. 重载（Overload）和重写（Override）的区别

维度 重载（Overload） 重写（Override）
发生位置 同一个类中 父子类之间
方法名 相同 相同
参数列表 必须不同（个数/类型/顺序） 必须相同
返回类型 可以不同 必须相同（或协变返回类型）
访问修饰符 可以不同 不能比父类更严格
抛出异常 可以不同 不能比父类抛出更宽泛的异常
编译时/运行时 编译时多态 运行时多态
static 方法 可以重载 不能被重写（static 方法属于类，不属于对象）

@Override 注解的作用：编译器校验是否真的重写了父类方法，防止手误（参数写错变成重载）。

5. 接口和抽象类的区别

维度 抽象类（abstract class） 接口（interface）
关键字 abstract class interface
继承/实现 单继承（extends 只能一个） 多实现（implements 可以多个）
构造方法 有 没有
成员变量 可以是普通变量 只能是 public static final 常量
方法 抽象方法 + 具体方法都行 JDK8+: 抽象方法 + default + static；JDK9+: 支持 private
访问修饰符 任意（public/protected/private） 方法默认 public，变量默认 public static final

如何选择？

场景 选择
多个类共享同一段代码/成员变量 抽象类
定义一种行为规范，不关心实现 接口（如 Runnable, Comparable）
需要多继承能力 接口
is-a 关系，有状态 抽象类
has-a / can-do 能力 接口

6. String、StringBuffer、StringBuilder 的区别

维度 String StringBuffer StringBuilder
可变性 不可变（final） 可变 可变
线程安全 安全（不可变天生安全） 安全（synchronized） 不安全
性能 拼接频繁时差（创建新对象） 中（有同步开销） 最快
适用场景 少量字符串操作 多线程环境 单线程环境

面试常问：为什么 String 要设计成不可变？

原因 说明
字符串常量池 可共享，节省内存
HashMap 的 key 不可变保证 hashCode 不会变，否则 get 不到
线程安全 多线程随便用，不用加锁
安全 类加载、数据库连接等关键参数不会被随意篡改

7. Java 异常体系

对比 受检异常（Checked） 非受检异常（Unchecked）
父类 Exception（非 RuntimeException） RuntimeException
编译器检查 必须 try-catch 或 throws 不强制
代表 IOException, SQLException NullPointerException
理念 外部因素导致，可以预期恢复 程序 bug，应该修代码

finally 一定会执行吗？

不一定。以下情况不会执行：

  • System.exit(0) 直接终止 JVM
  • 守护线程退出
  • 死循环或死锁导致线程卡住

8. ArrayList 和 LinkedList 的区别

维度 ArrayList LinkedList
底层结构 动态数组 Object[] 双向链表
随机访问 **O(1)**，直接下标 **O(n)**，需遍历
头插/中间插入 **O(n)**，需移动元素 **O(1)**，改指针
内存占用 仅数据（连续内存） 数据 + 前后指针（每个节点多 2 个引用）
实现接口 List + RandomAccess List + Deque（双端队列）
扩容 1.5 倍扩容，数组复制 不需要，链式添加

选哪个？ 大量读/遍历 → ArrayList；大量头插/删除 → LinkedList；两项都有 → 优先 ArrayList（内存友好，CPU 缓存友好）。

9. HashMap 的底层原理

数据结构

JDK 1.7：数组 + 单向链表
**JDK 1.8+**：数组 + 链表 + 红黑树（链表长度 ≥ 8 时转为红黑树，≤ 6 时退化为链表）

put 的完整流程

  1. 对 key 求 hashCode()，再通过 (n-1) & hash 计算桶下标
  2. 该桶为空 → 直接插入
  3. 该桶非空 → 判断 key 是否相同（hash 值相等 && equals 相等）
    • 相同 → 覆盖旧值
    • 不同 →
      • 如果是红黑树节点 → 走树插入
      • 如果是链表节点 → 尾插（JDK 1.8）或头插（JDK 1.7）
        • 遍历过程中发现相同 key → 覆盖
        • 遍历到末尾仍未找到 → 插入，链表长度 ≥ 8 且数组长度 ≥ 64 → 转红黑树
  4. size++，判断是否超过阈值 → resize() 扩容

关键参数

参数 默认值 说明
初始容量 16 必须是 2 的幂（方便位运算取模）
负载因子 0.75 超过容量 × 负载因子时扩容
树化阈值 8 链表长度达到 8 转红黑树
退化阈值 6 红黑树节点数 ≤ 6 退化为链表
最小树化容量 64 数组长度 < 64 时优先扩容而非树化

为什么容量是 2 的幂？

(n-1) & hash 代替 hash % n，位运算比取模快。只有 n 是 2 的幂时，(n-1) & hash 才等价于取模且分布均匀。

JDK 1.7 vs 1.8 区别

维度 JDK 1.7 JDK 1.8
结构 数组 + 链表 数组 + 链表 + 红黑树
插入方式 头插法（resize 可能导致死循环） 尾插法（解决死循环问题）
hash 扰动 4 次位运算 + 4 次异或 1 次移位 + 1 次异或
扩容时机 size ≥ threshold && 当前桶非空 仅 size > threshold

10. Java 是值传递还是引用传递？

答：Java 只有值传递，没有引用传递。

参数类型 传递的是什么 方法内修改
基本类型 值的副本 不影响原变量
引用类型 引用的副本（指向同一对象） 修改对象内容会影响原对象；但修改引用本身（new 新对象）不影响原引用

一句话总结：Java 传递的是值的副本。基本类型传值的副本，引用类型传引用地址的副本。所以能改对象内容，但改不了引用指向。

11. hashCode 与 equals 的关系

一句话

equals 相等，hashCode 必须相等；hashCode 相等，equals 不一定相等。

三大铁律（来自 Object 源码文档）

规则 说明
自反性 同一对象多次调用 hashCode()，返回值必须相同（前提是 equals 用到的字段没变）
等价性 两个对象 equals 相等 → hashCode() 必须相等
非等价性 两个对象 equals 不等 → hashCode() 不要求不等（允许哈希冲突）

12. 深拷贝和浅拷贝的区别

定义

类型 行为
浅拷贝 复制对象本身，但对象内部引用类型字段仍指向同一个对象
深拷贝 复制对象本身，且递归复制所有引用类型字段，二者完全独立

实现深拷贝的三种方式

序列化 ByteArrayOutputStream + ObjectOutputStream 自动递归，代码简单 性能差，要求 Serializable
JSON 序列化 Jackson/Gson 序列化再反序列化 简单，不用实现接口 性能一般，依赖第三方库

面试要点

问题 答案
Object.clone() 是深还是浅？ 浅拷贝，只复制对象本身
Arrays.copyOf() 呢？ 数组元素是引用类型时，也是浅拷贝
深拷贝和不可变对象的关系？ String、Integer 等不可变对象，浅拷贝就够了，反正内容改不了
什么场景用深拷贝？ 需要修改副本且不影响原对象时（如缓存返回、配置副本）
